适用范围：`services/business/**` 总体架构、Kitex server、领域分层、Go package、依赖方向、本地启动和架构测试  
相关代码路径：`services/business/**`、`api/thrift/business/**`、`db/migrations/iterations/**/business/**`、`tests/business/**`

## 目标

定义业务微服务的代码结构和架构边界。业务服务作为业务事实源，承接前端 API 适配层和 Agent 微服务的业务 RPC 调用，负责权限、事务、业务数据库、错误码和审计。

## 非目标

- 不实现 Eino Agent、Graph、Workflow、TurnLoop、AG-UI 或 Agent DB。
- 不直接暴露数据库表 CRUD 给 Agent 或前端。
- 不把平台后台、普通用户端和公开页面的前端状态放入业务服务。

## 依赖方向

| 层 | 可依赖 | 禁止依赖 |
| --- | --- | --- |
| `transport/rpc` | application、DTO mapper、errors | repository、GORM model、前端模型、Agent Runtime |
| `application/*` | domain、repository interface、transaction、idempotency、audit | Gin/Kitex handler 细节、GORM 具体实现 |
| `domain/*` | domain common、value object | transport DTO、repository SQL、外部 RPC |
| `infra/repository` | domain repository interface、GORM | application 业务编排 |
| `infra/*` | config、logger、crypto、tos、transaction | domain 规则 |

## 设计模式

| 模式 | 使用位置 | 目的 |
| --- | --- | --- |
| Application Service | `internal/application/<domain>` | 编排权限、事务、幂等、repository 和审计 |
| Domain Service | `internal/domain/<domain>` | 表达业务规则、状态机和校验 |
| Repository | `internal/infra/repository/postgres` | 封装 GORM 数据访问，不承载业务规则 |
| Policy / Specification | 权限、Tool 策略、Skill 发布校验、项目访问 | 让复杂规则可组合、可测试 |
| Transaction Script | 积分冻结、资产保存扣费、企业转让、作品分享 | 清晰控制事务边界 |
| Idempotency Guard | 所有写操作 | 防止重复创建、重复冻结、重复扣费 |
| Audit Decorator | 写操作 application service | 统一审计字段、结果和脱敏 |

## 服务启动流程

1. 读取 `.env.local` 或环境变量。
2. 初始化结构化 logger，生成 `service=business`、`env`、`instance_id`。
3. 初始化 PostgreSQL GORM client，禁用 AutoMigrate。
4. 初始化 etcd registry，注册 Kitex business service。
5. 初始化 TOS client、crypto provider、id generator、clock。
6. 装配 repository、application service、RPC handler。
7. 启动 Kitex server，配置超时、panic recover、日志和 trace middleware。

## 通用 RPC handler 规则

| 步骤 | 说明 |
| --- | --- |
| 1. 参数绑定 | 只做 Thrift DTO 到 application input 的转换 |
| 2. 上下文提取 | 读取 `AuthContext`、`RequestMeta`、trace_id |
| 3. 调用 application | 不在 handler 写业务规则 |
| 4. 错误映射 | 将 domain/application 错误转为稳定业务错误码 |
| 5. 响应转换 | 返回场景 DTO，不返回 ORM 对象 |
| 6. 日志 | 记录 rpc_method、trace_id、operator、latency、error_code |

## 通用 HTTP API handler 规则

业务服务默认以 Kitex 为主；因产品已明确 `frontend/**` 和 `admin_frontend/**` 需要业务 API 适配层，业务服务可使用 Gin 暴露最小 HTTP transport。HTTP handler 不新增业务规则，只把前端请求转换为 application input 或内部 RPC DTO。

| 分组 | 路径前缀 | 调用方 | 鉴权来源 | 说明 |
| --- | --- | --- | --- | --- |
| Public API | `/api/public/**` | 公开页、未登录访客 | anonymous 或登录态可选 | 只返回公开快照、公开 Skill 摘要和非敏感展示字段。 |
| User API | `/api/**`，排除 `/api/admin/**` | 用户端 `frontend/**` | 用户登录 cookie/token | 当前空间由服务端登录态解析，不信任前端自造空间。 |
| Admin API | `/api/admin/**` | 管理端 `admin_frontend/**` | 后台管理员独立 cookie/token | `RequestMeta.source=admin`，写操作必须携带原因并审计。 |

HTTP handler 步骤：

| 步骤 | 说明 |
| --- | --- |
| 1. 解析登录态 | 用户端生成 `AuthContext.login_identity_type=personal/enterprise_member`；后台生成 `admin`；公开读为 `anonymous`。 |
| 2. 生成 RequestMeta | 从 `X-Request-ID`、`X-Trace-ID`、`Idempotency-Key`、来源端生成统一 `RequestMeta`。 |
| 3. 参数绑定和白名单校验 | Gin binding 只做类型、必填、长度、枚举和分页上限校验。 |
| 4. 调用 application | 不在 handler 访问 repository，不复制 domain 规则。 |
| 5. 错误映射 | `BusinessError` 映射到 HTTP status 和稳定 `error.code`。 |
| 6. 响应包装 | 返回场景 DTO，不返回 ORM、内部成本、密钥、Prompt、推理链路。 |
| 7. 日志 | 输出 `http_method`、`path_template`、`source`、`trace_id`、`latency_ms`、`error_code`。 |

## 本地配置

| 配置项 | 说明 | 示例 |
| --- | --- | --- |
| `BUSINESS_DATABASE_URL` | 业务 PostgreSQL 连接 | `postgres://dora:password@localhost:5432/dora_business?sslmode=disable` |
| `ETCD_ENDPOINTS` | Kitex 服务发现 | `localhost:2379` |
| `BUSINESS_SERVICE_NAME` | Kitex 服务名 | `dora.business` |

## 日志基础字段

所有 RPC 请求至少输出：

| 字段 | 来源 |
| --- | --- |
| `trace_id` | `RequestMeta.trace_id` |
| `request_id` | `RequestMeta.request_id` |
| `service` | 固定 `business` |
| `rpc_method` | Kitex method |
| `operator_type` / `operator_id` | `AuthContext` |
| `tenant_id` / `space_id` / `enterprise_id` | `AuthContext` |
| `business_action` | application 设置 |
| `resource_type` / `resource_id` | application 设置 |
| `idempotency_key` | 写操作 |
| `error_code` | 失败时 |
| `latency_ms` | middleware |

## 【Agent开发】需要对接的架构能力

| 【Agent开发】能力 | 业务服务入口 | Agent 需要提供参数 | 业务服务返回 | 约束 |
| --- | --- | --- | --- | --- |
| 业务 RPC 调用 | Kitex client -> business server | `AuthContext`、`RequestMeta`、业务 DTO | 业务 DTO 或 `BusinessError` | Agent 不直接访问业务 DB |
| 统一 trace | 所有 RPC | `trace_id`、`request_id`、`source=agent_service` | 响应带同一 `trace_id` | 日志和审计贯通 |
| 幂等写 | 所有写 RPC | `idempotency_key`、稳定请求参数 | 同一业务结果或 `IDEMPOTENCY_CONFLICT` | Agent 重试必须复用同一 key |
| 权限上下文 | 所有私有读写 RPC | 当前登录身份、空间、企业角色 | 权限结果或拒绝错误 | 前端传入空间不能作为唯一信任源 |

## 架构测试

- RPC handler 不直接引用 GORM model。
- repository 不包含业务状态机跳转逻辑。
- migration 中不出现数据库级外键或引用约束关键字。
- application 写操作都接收 `RequestMeta.idempotency_key`。
- 所有列表查询 input 都包含分页，默认 `page_size=10`。
- 日志脱敏测试覆盖 API Key、兑换码、TOS 签名、私密创作内容。
