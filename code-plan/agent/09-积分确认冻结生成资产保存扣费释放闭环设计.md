适用范围：生成类任务和直接平台 Tool 调用的安全评估、积分预估、人工确认、冻结、执行、资产保存、扣费和释放

后续实现落点：`api/thrift/business_agent_service.thrift`

## 文档目标

- 定义生成任务和直接平台 Tool 调用的 Agent 侧职责内闭环。
- 固化安全评估、积分预估、确认、冻结、生成、资产保存、扣费或释放的顺序。
- 固化 Tool 计费项从预估、冻结、执行结果到实际扣费的 `estimate_item_id` 关联关系。
- 明确积分账户、冻结、扣减、流水等业务事实由业务微服务维护。
- 处理失败、取消、超时、部分完成和资产保存失败。

## 主链路

```text
用户输入
  -> 安全评估
  -> 模型与参数锁定
  -> 积分预估（生成资源和 Tool 用量）
  -> 人工确认
  -> 冻结积分
  -> 执行生成 Tool
  -> 准备生成产物 TOS 对象并上传
  -> 保存资产与最终资产元素
  -> 保存成功则确认扣费
  -> 保存失败或未完成则释放冻结积分
  -> 输出 AG-UI 结果和快照
```

直接平台 Tool 调用分支：

```text
用户输入
  -> Tool 意图识别和开放能力校验
  -> 安全评估
  -> Tool 用量预估
  -> 人工确认
  -> 冻结积分
  -> 执行 Tool
  -> 执行成功则按实际用量扣费
  -> 执行失败、取消或超时则释放冻结积分
  -> 输出 AG-UI 结果和快照
```

## 职责内闭环必须明确

Agent 侧负责闭环：

- 固定流程顺序：安全评估 -> 预估 -> 确认 -> 冻结 -> 生成或 Tool 执行 -> 保存资产或结算 Tool -> 扣费/释放。
- 保存 Runtime 状态、interrupt、task、tool_call、event、artifact 引用。
- 将业务 RPC 结果映射为 AG-UI 事件。
- 在失败、取消、归档阻断时调用释放冻结积分能力。
- 保存 `estimate_id`、`estimate_item_id`、`freeze_id` 与 `tool_call_id` / `artifact_id` 的运行态关联，防止重复扣费。

Agent 侧不负责闭环：

- 不维护积分账户、积分批次、积分流水。
- 不保存最终资产事实、最终资产元素事实、项目资产关系事实。
- 不决定企业积分权限和成员消耗明细。

## 资源计费输入

| 资源类型 | 计费输入 | Agent 校验 | 业务计价依据 |
| --- | --- | --- | --- |
| `image` | `model_id`、`pricing_snapshot_id`、`quantity`、`size`、`quality` | 数量上限、尺寸枚举、模型可用性 | 模型单价、数量、规格。 |
| `music` | `model_id`、`pricing_snapshot_id`、`duration_seconds`、`vocal_mode`、`quantity` | 时长范围、模式枚举 | 时长、数量、模型单价。 |
| `video` | `model_id`、`pricing_snapshot_id`、`duration_seconds`、`resolution`、`fps`、`quantity` | 时长和分辨率上限 | 时长、清晰度、数量、模型单价。 |
| `tool_usage` | `tool_name`、`tool_type`、`billing_unit`、`quantity`、`metadata_summary` | Tool 开放策略、风险等级、单位枚举、用量上限 | Tool 计价策略、实际用量、免费策略。 |

Agent 不计算最终价格，只把业务要求的计费输入传给 `CreditService.EstimateGenerationCredits` 或 `CreditService.EstimateToolCredits`。前端展示的积分数以业务响应为准。生成类 Tool 产生的可见资产在 `CommitGeneratedAssetAndCharge` 中随 `artifacts[].estimate_item_id` 结算；不产生业务资产的直接 Tool 调用在 `ChargeToolUsageCredits` 中随 `charge_items[].estimate_item_id` 结算。

## 事件顺序

正常分支：

```text
safety.prompt.evaluating
  -> safety.prompt.evaluated
  -> credits.estimated
  -> confirmation.required
  -> confirmation.accepted
  -> credits.frozen
  -> tool.call.started
  -> tool.call.progress
  -> tool.call.completed
  -> generation.progress
  -> generation.artifact.completed
  -> asset.save.started
  -> 生成产物上传到业务签发 TOS object key
  -> asset.save.completed
  -> workspace.assets.updated
  -> credits.charged
  -> process.snapshot.saved
  -> agent.run.completed
```

直接 Tool 成功分支：

```text
safety.prompt.evaluating
  -> safety.prompt.evaluated
  -> credits.estimated
  -> confirmation.required
  -> confirmation.accepted
  -> credits.frozen
  -> tool.call.started
  -> tool.call.progress
  -> tool.call.completed
  -> credits.charged
  -> process.snapshot.saved
  -> agent.run.completed
```

失败分支：

| 分支 | 事件 |
| --- | --- |
| 安全阻断 | `safety.prompt.blocked` -> `agent.run.failed` |
| 积分不足 | `credits.insufficient` -> `agent.run.failed` |
| 用户拒绝 | `confirmation.rejected` -> `agent.run.cancelled` |
| 冻结失败 | `credits.freeze.failed` -> `agent.run.failed` |
| 生成失败 | `tool.call.failed` -> `credits.released` -> `agent.run.failed` |
| 部分完成 | `generation.partial.completed` -> `asset.save.completed` -> `credits.charged` + `credits.released` |
| 保存失败 | `asset.save.failed` -> `credits.released` -> `agent.run.failed` |

## 测试

| 场景 | 断言 |
| --- | --- |
| 正常完成 | 安全、预估、确认、冻结、生成、保存、扣费事件顺序完整。 |
| 重复确认 | 同一幂等键只冻结一次。 |
| 积分不足 | 不创建冻结，不执行 Tool。 |
| Tool 失败 | 释放冻结积分，run failed。 |
| 直接 Tool 成功扣费 | `ChargeToolUsageCredits` 只扣实际完成 `charge_items[]`，重复 `tool_call_id + estimate_item_id` 不重复扣费。 |
| 生成 Tool 用量扣费 | `CommitGeneratedAssetAndCharge` 使用 `artifacts[].estimate_item_id` 返回 `charged_line_items[]`。 |
| 部分完成 | 完成资产保存扣费，未完成释放。 |
| 保存失败 | 不扣费或按业务返回释放，错误事件可见。 |
| 项目归档 | commit 前二次校验失败并释放。 |

## 业务开发对齐点

- 资产保存和扣费是否需要同一个业务事务能力。
- 部分完成时每个 asset 如何计算扣费和释放。
- 冻结后过期积分的业务处理由业务服务如何返回。
- 生成取消时已完成且保存成功资产的扣费口径。
