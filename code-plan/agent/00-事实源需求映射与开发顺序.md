## 契约先行顺序

1. `ProjectService.CheckProjectAccess`、`AccountSpaceService.ResolveCurrentSpaceContext`、`AssetService.BatchCheckAssetAccess` 先确认，因为它们影响 session/run 创建和权限闭环。
2. `SkillCatalogService`、`ToolCapabilityService`、`ModelConfigService` 再确认，因为它们影响 Skill 路由、Tool 策略和模型选择。
3. `CreditService`、`AssetCreditCommitService` 再确认，因为它们影响扣费事务和失败补偿。
4. AG-UI 事件 payload 和 Agent API DTO 随上述 RPC 确认同步冻结。
5. Agent DB migration 在 API/RPC/AG-UI 字段冻结后生成，避免字段反复漂移。
6. 执行 [14](./14-生产级闭眼开发门禁与任务切片验收设计.md) 的设计门禁检查；代码级 IDL、OpenAPI、AG-UI schema、migration 和 fixture 在对应功能切片内先补契约再编码。

## 业务开发对齐点

- 哪些业务能力必须先落 RPC 契约。
- 哪些字段属于业务事实，Agent 只能保存引用。
- 项目、积分、资产、作品和权限的最终解释权归属。
- Agent 侧是否需要额外业务批量查询能力来避免逐条 RPC。
- 所有业务 RPC 是否统一接收 `auth_context`、`request_meta` 和稳定错误结构。
- 写操作是否均支持 `idempotency_key`，并在参数不一致时返回 `IDEMPOTENCY_CONFLICT`。
- 安全证据 `safety_evidence` 是否由业务服务保存摘要并校验过期时间、场景和内容摘要。
