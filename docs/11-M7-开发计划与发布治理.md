## 2. 目标

1. 补齐 M7 运营治理数据底座、服务模块和管理端页面，形成可灰度、可观测、可回滚的发布闭环。
2. 完成 M6 用户端剩余体验（真实数据接入、市场前台、创作者后台、费用披露组件）。
3. 用本仓库的回归门禁（`make regression` + CI Regression Gates）作为每个批次的准入条件。

非目标：不新增创作能力；不修改已发布 SkillVersion 内容；不在本阶段引入真实模型供应商。

## 4. 发布门禁（Release Gates）

每个批次进入下一步前必须通过以下门禁：

1. **Contract Gate**：`go test ./internal/contracts/...` 全部通过；`tests/fixtures/contracts/**` 与 Go 校验器一致；StateEnumRegistry 与 `state.go` 枚举一致。
2. **Migration Gate**：所有 migration 有 up/down 配对，dry-run 与回滚验证通过，禁止数据库级外键；回填任务写入 `migration_jobs` 并出校验报告。
3. **Fixture Gate**：`scripts/validate-fixtures.py` 与 `tests/e2e/*/scenarios.json` 索引校验通过，8 个必需 E2E 夹具齐全且契约引用存在。
4. **Fake Provider Gate**：`tests/e2e/fake-provider/**` 六种行为（deterministic_success、async_pending、partial_success、transient_failure、terminal_failure、slow_callback）全部可执行且幂等。
5. **Feature Flag Gate**：新能力默认关闭；打开真实供应商前必须人工审批。
6. **Observability Gate**：§6 指标可查询、可告警，关键路径日志带 trace_id / run_id / tool_call_id。
7. **Rollback Gate**：§7 回滚预案演练通过，回滚不破坏账务与审计。

## 5. Feature Flags

| Flag | 默认 | 控制范围 |
| --- | --- | --- |
| `agent_runtime_v2` | off | Agent Router v2 入口、Graph/Board Runtime、AG-UI replay writer |
| `tool_generation_v2` | off | ToolPlan 确认、积分冻结/提交、生成任务与资产提交 |
| `marketplace_v2` | off | 市场浏览、安装、Skill 使用费与结算 |
| `creator_portal_v2` | off | 创作者发布后台 |
| `admin_governance_v2` | off | 管理端发布治理与市场治理页面 |

放开顺序：`agent_runtime_v2` → `tool_generation_v2` → `marketplace_v2`（免费）→ `marketplace_v2`（付费）→ `creator_portal_v2` / `admin_governance_v2`。

## 6. 观测指标

| 指标 | 含义 | 告警口径 |
| --- | --- | --- |
| `agent_run_success_rate` | run 成功率 | 5 分钟窗口 < 95% |
| `router_decision_latency_ms` | Router P95 延迟 | P95 > 3000ms |
| `board_patch_replay_error_count` | Board patch replay 失败数 | > 0 即告警 |
| `graph_resume_failure_count` | Graph 中断恢复失败数 | > 0 即告警 |
| `tool_task_success_rate` | 生成任务成功率 | 5 分钟窗口 < 90% |
| `credit_freeze_leak_count` | 冻结泄漏（超时未提交/释放） | > 0 即告警 |
| `skill_usage_charge_error_count` | Skill 使用费扣费错误数 | > 0 即告警 |
| `marketplace_install_failure_count` | 市场安装失败数 | 突增告警 |
| `settlement_reverse_count` | 结算 reversal 次数 | 突增人工复核 |

## 7. 回滚预案

触发条件：任一告警级指标越界、扣费争议、资产丢失风险或安全事件。

回滚动作按顺序执行：

1. 关闭对应 feature flag（`agent_runtime_v2` / `tool_generation_v2` / `marketplace_v2`），新 run 立即回到旧入口或停止创建。
2. worker 停止消费生成队列新任务，已 inflight 任务允许收敛到终态。
3. 释放所有未进入 `asset_commit_started` 的积分冻结（credit hold 与 skill usage freeze），保证 `credit_freeze_leak_count` 归零。
4. 保持 AG-UI replay 可读：历史 run 的事件流按 seq 与 dedupe_key 原样可回放，断线客户端可凭 Last-Event-ID 恢复；重复投递事件必须被 dedupe_key 幂等忽略。
5. 暂停受影响 listing（不修改 SkillVersion 内容），必要时进入退款仲裁链路。
6. 登记 `operational_incidents`，产出复盘；回滚过程不删除任何积分账务、skill usage、ledger、资产与审计记录。

## 9. 风险

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| `api/**` 契约源已删除 | 无法重新生成 Kitex/OpenAPI 代码，前后端契约漂移 | 短期以 `kitex_gen/**` 与 Go 契约包为事实源；重构 RPC 前先恢复 thrift 定义 |
| 治理表缺失期间发生事故 | 无法追溯发布批次与影响面 | R1 数据底座优先级最高，先于任何灰度 |
| 前端 mock 数据上线 | 用户看到假数据 | R5 完成前用户端保持 flag off |
| 回滚演练缺失 | 真实事故时回滚不可用 | R6 强制演练并留档 |

## 10. 验收标准

- [ ] R1-R6 全部批次按 §4 七道门禁通过。
- [ ] 治理表可写入并可按 run_id / batch_id / incident_id 追溯。
- [ ] 管理端五个治理页面可用且高风险操作有确认与审计。
- [ ] 用户端工作台、市场前台、创作者后台脱离 mock 数据。
- [ ] 回滚演练证明：关闭 flag 后新 run 停止、冻结释放、AG-UI replay 可读、账务与审计零删除。
