## 1. 核心范式

〔原话〕**"工具是死的，编排是活的。"**

- 一批预定义工具，核心能力固定，不凭空造新工具。
- "调哪些、什么顺序、串行还是并行、走哪条内部路径"由 agent 每次按需求+输入+skill 动态决定。
- 〔原话〕类比乐高：积木块固定，造型无数，"我负责根据您的目标决定怎么拼、拼什么顺序"。

## 2. 一次请求的运转流程

1. **判断任务复杂度**〔原话〕：简单单步（一首歌/一张图）→ 直接调对应工具，无需故事板；短视频（≤15s）→ 可直接生成跳过完整流程；复杂项目（MV/短片/广告）→ 完整流程。
2. **匹配并加载 skill**：复合需求自动加载对应 skill；一次只激活一个。
3. **建规格** `Final_Video_Spec.md`（类型/画幅/时长/视觉风格/语言/模型偏好）。
4. **按 skill 骨架逐阶段**：分析素材 → 故事板 → 元素参考图 → 每镜头关键帧 → 逐镜头视频（演唱=对口型、叙事=多模态）→ 合成。
5. **关键节点暂停确认**：每阶段后停下，卡片 + 候选选项（芯片，可分页）。
6. **合成导出**。

**动态编排实证**：输入"帮我做一首关于暗恋的中文流行歌，配一支竖屏MV" → agent 判定"歌曲创作+MV 复合项目" → 动态加载「音乐MV制作技能」（卡片显示 Skill已完成）→ 建规格（9:16；关键帧 Nano Banana / 对口型 OmniHuman 1.5 / 动态镜头 Seedance 2.0）→ 弹视觉风格 4 候选芯片（清新日系胶片/都市夜景写实/唯美插画动漫/电影感文艺写实，带 1/2 分页）暂停等选择。

**同一目标不同编排**〔原话〕：只有一句话描述→直接文字生视频一步完成；有参考图→图→参考图→多模态生视频；有剧本+角色→故事板→角色参考图→逐镜头→合成；有音乐做MV→故事板→角色图→对口型→合成。

## 3. Skill 机制

### 3.1 skill 是一份可编辑文档

项目内 `skill.md` 分区（音乐MV skill 实测结构）：
- **Skill 调用规则**（91/200 字符）：告诉 AI 何时该调用此 skill（多 skill 开启时的路由依据）。
- **流程规划**：阶段列表 + 每步用什么工具 + **依赖关系**（`2→1；3→1,2；4→2；5→3；6→3,5；7→3,6`）+ **何时暂停**（"不要一次性运行所有步骤，在每个关键阶段停止与用户确认"）+ 工作流修正规则（缺驱动音频→转音频生成；用户坚持无音频生成→必须警示唇形同步精度下降，获确认才继续）。
- **素材分析**：分析后要持久化什么（BPM、强拍、分段标签、歌词行 start_ms/end_ms；下游镜头必须引用这些 ID/时间戳）。
- **故事板设计**：关键元素怎么设计（element_id 规划、角色多外观声明、场景描述方位、用户资产描述必须与素材一致）；镜头怎么写（场景+元素+镜头语言）；MV 时间规划核心规则（镜头内歌词必须是完整单元、逐行时间戳 ±1s 缓冲、叙事+对口型组合，副歌高潮用对口型）。
- **媒体生成**：分步模型偏好（元素图 TextToImage/Nano Banana 2/2K/角色三视图；关键帧 ImageToImage/Nano Banana Pro/2K/参考元素图+前序最相似关键帧；叙事镜头 MultiModalToVideo/Seedance 2.0/720p；对口型 ImageToVideoByAudio/OmniHuman 1.5/1080p）。
- **提示词写法**：中文环境用中文写 prompt（旁白语言除外，遵循 Spec 的 Output Language）；图像提示词六要素（风格锚点/取景/灯光/调色/渲染/情绪潜台词）；关键帧分起始帧/结束帧/高光帧的取帧法；视频提示词只描述"变化与动态"不重复静态；对口型模板（镜头运动+情绪+说话状态+动作）；固定负面约束（no music / no subtitles / 有独立旁白轨则视频 prompt 不写旁白词）；**安全合规重写顺序**（名人→虚拟演员、版权IP→去标识保留特征、品牌→原创同型、血腥/裸露/恐怖→柔化替换）与**六大禁区直接拒绝**（政治敏感/极端主义/毒赌诈/假新闻/未成年不宜等）。
- **视频剪辑**：静音 video 轨；按时间戳对齐音乐时间线；对口型片段不得变速不得改时间戳；空隙用叙事片段填补；±1s 缓冲与故事板同规则。

### 3.2 双副本与编辑语义〔原话〕

- 两份：技能注册表里的**原始模板** + 当前项目里的**可编辑副本**（skill.md，"活的项目副本"）。
- 编辑**只影响当前项目**，不影响其他项目、不改注册表模板。
- **下一条消息起生效**；**已完成的步骤不回滚**（"已执行的步骤不会回滚……从当前位置按新流程继续"）。
- 若改的是"已执行的规则"（如已过的暂停点），已过去的不再生效，只影响未执行的后续步骤。
- ⚠️"重新加载 skill"会用注册表模板**覆盖**项目内编辑，改动丢失（agent 主动警示）。

### 3.3 skill 选择与裁决〔原话〕

- **一次只加载一个 skill**。依据"Skill 调用规则"字段做需求匹配。
- 需求明显偏向其一 → 直接加载匹配度更高的，回复里说明理由。
- 模糊 → 出候选让**用户裁决**。
- 不属于任何 skill → 不加载，直接执行单步任务。
- 需要多个 skill 的能力 → "把需求告诉我，我来判断用哪个 skill 作为主框架，并在执行中灵活补充另一套逻辑"——**可以手动读取其他 skill 文档内容参考**其中规则风格，但不是"同时加载"，不硬拼两个。

### 3.4 Skill 定骨架，Agent 填血肉〔原话，分工表〕

| 层面 | Skill 负责 | 动态决策负责 |
|---|:---:|:---:|
| 阶段顺序（1→2→3…） | ✅ 定死 | |
| 暂停节点位置 | ✅ 定死 | |
| 工具选择（大类） | ✅ 定死 | |
| 具体内容决策（节奏/情绪/色调） | | ✅ |
| 用户素材处置（有→绑定不重生成；无→生成） | | ✅ |
| 异常与冲突处理（生成失败/中途改需求） | | ✅ |
| 参数匹配（哪个模型适合这首歌的情绪） | | ✅ |
| 任务边界判断（要不要加载 skill / 单步够不够） | | ✅ |

- **无 skill 也能编排**〔原话〕："可以，但质量和可靠性不同"——按任务复杂度+工具依赖通用规则+内置生成顺序自行推导，但缺领域经验（如"唇形同步要等驱动音频确认"）、暂停点判断不精准。**"Skill 本质上是把有经验的制片人判断提前固化下来。"**

## 4. 工具体系

### 4.1 六大类（按职责边界，互不越界）〔原话〕

| 大类 | 工具 | 职责 | 权限 |
|---|---|---|---|
| 规划类 | `storyboard_designer` | 创建/修改故事板蓝图（镜头结构、元素定义、音频轨规划） | 只管规划，不生成媒体 |
| 生成类 | `media_generator` | 生成所有媒体资产（图/视频/旁白/音乐/歌曲）；管理资产生命周期；绑定到故事板 | **唯一资产写权限** |
| 分析/准备类 | `resource_prepare_and_analyze` | 理解已有素材：内容理解/关键帧提取/音轨分离/结构分解 | 只处理已有素材，不创作 |
| 合成类 | `video_assembler` | 按故事板时间线拼片段+混音，输出成片 | 只组装，不生成新内容 |
| 查询类 | `query_assets_info` / `query_resource_generation_info` / `query_media_prompt_draft` / `resource_content_analyzer` | 资产状态/版本/绑定/提示词草稿/媒体内容视觉分析 | **只读** |
| 辅助类 | `search` / `skill` | 外部检索 / 技能加载 | — |

### 4.2 skill↔tool 挂钩方式〔原话〕

- **Skill 直接点名工具"大类"**："分析音乐→resource_prepare_and_analyze""生成故事板→storyboard_designer""生成媒体→media_generator"——阶段级指引。
- **具体路径由 agent 运行时判断**：media_generator 内部有十几条路径（TextToImage / ImageToImage / TextToVideo / FirstFrameToVideo / MultiModalToVideo / VideoInterp / ImagesToVideo / ImageToVideoByAudio / Text-to-Narration / text_to_instrumental / lyrics_to_song / super_resolution），选哪条"是我根据当时的实际情况动态决定的，skill 不管这层"。
- 场景硬规则例外：数字人对口型**务必**用 ImageToVideoByAudio，其他工具嘴型同步效果差（skill/系统层强调）。

### 4.3 串并行〔原话〕

- 工具间有**严格先后顺序约束**：故事板（蓝图）→ 元素参考图（一致性锚点）→ 镜头视频（依赖参考图）→ 合成（依赖所有素材）。
- **无依赖可并行**：多个独立镜头视频、多个角色参考图、旁白与视频、VO+BGM。
- 判断"哪些并行哪些串行"由 agent 依据依赖关系动态定。

### 4.4 数据传递：三层绑定〔原话+逆向〕

- **工具互不调用，agent 是调度枢纽**："负责在每次派发任务时，把上一步产出的 resource_id / asset_id 作为下一步的参考输入……工具本身不互相调用，是我在中间做传递路由。"
- 三层：`storyboard(element_id / shot_id / audio_layer)` → `asset(asset_id，逻辑资产，版本/favorite)` → `resource(resource_id，物理文件 @ GCS)`。
- 流转示例：storyboard_designer 产 shot_id → media_generator 生成产 resource_id、注册 asset_id、绑到 shot_id → 下一步 media_generator 从绑定关系取参考 resource_id → video_assembler 按 shot_id 绑定的视频 resource 顺序拼接。
- resource 用语义 id（`Element_*` / `Shot_*` / `Audio_*`）〔逆向〕。

## 5. 权限模型（双层）〔原话〕

**只有 media_generator 有资产写权限**，且写行为受明确约束：

| 操作 | 允许？ | 条件 |
|---|:---:|---|
| 生成新 asset 并绑定故事板 | ✅ | 正常生成流程 |
| 给已有 asset 添加新版本（重新生成） | ✅ | 重新生成时，旧版本不删除，新版本并列 |
| 更换 asset 绑定的"首选版本" | ✅ | 用户选了哪个版本，就把它标记为 favorite |
| 删除某个 asset | ✅ | 需要明确指令，不会自动删 |
| 覆盖/删除用户上传的原始文件 | ❌ 绝对不行 | 上传素材永久保留 |
| 未经授权替换已确认的资产 | ❌ 绝对不行 | 见"已确认资产"保护规则 |

**"已确认资产"保护**：用户对话确认过的资产（"这个版本好，用这个"）后续流程不再改动；重新生成只产并列新版本；替换已确认版本必须经明确指令。

**权限分两层**：
- **物理层 = 系统定死**："上传文件只读""查询工具不能写""重新生成不自动覆盖已确认版本"——"这些是架构硬约束，skill 无法改变，我也无法绕过。"
- **行为层 = agent（调度层）执行**："不擅自重做已确认资产""合规拦截后停下来问您"——是 agent 的操作规范，skill 可在一定范围内影响（如 skill 可以规定不同的暂停节奏）。

## 6. 交互语义（九轮追问实录要点）

1. **卡点上用户能做什么**〔原话〕：不只确认/打回——可**精改单个镜头**、**跳过某阶段**、**回退上一步重做**、**部分重做**；两阶段并行"有条件支持"（取决于依赖，非主观决定）。**"卡点的位置是 skill 定的；卡点上你的操作权限是完全开放的……并行的边界取决于依赖关系，我会提前告知风险，最终由您决定。"**
2. **中途全局改需求**（做到关键帧时说"整体改日系动漫"）〔原话〕：**不擅自改已确认内容** → 先问范围（想改哪些）→ 影响评估并告知（哪些已生成内容受影响、哪些**可复用**——如场景纯环境镜头风格改动小、重做成本多少）→ 用户授权 → **最小化返工**（元素图=风格核心须重做；关键帧视情况——构图/人物/镜头景别不变可保留作**构图参考**重绘；故事板描述更新 `Final_Video_Spec.md` 视觉风格字段作后续新基准）。**"已确认的内容，我不擅自改。"**
3. **单镜头不满意只重做它**〔原话〕：每个镜头是**独立资产单元**——只重做 Shot_3（参考原关键帧+音频保持一致性），**不重跑整个流程、不动其他镜头、不重置时间线**；可一次生成 2~3 个备选变体（共享关键帧与音频锚点，差异在动作/镜头运动/细节表现）供挑选后替换进时间线。
4. **用户上传素材**〔原话〕：**上传即分析**（识别特征/用途）→ 指定为角色则**绑定为关键元素直接使用**，"不擅自重画您的素材"；质量有隐患（背景干扰/分辨率低/画风与视频模型适配差）会**告知问题并给选择**（直接用 / 生成一版"更适合模型输入的优化版"），原图与优化版都保留可切换；**素材与 spec 风格冲突 → 主动问，"您的意图优先于 spec 里的预设"**。
5. **失败与合规**〔原话〕：技术失败（生成质量/超时）→ **自主调整提示词/参数重试**，不必打扰；**合规拦截 → 绝不自动改写绕过**——"改写后的内容可能偏离您的原意……您有权知道哪里踩了红线"——告知被拦原因 → 提出调整方向 → **等您授权后再重写重试**；**单镜头失败不卡整条流程**：独立镜头继续跑，依赖该镜头的后续步骤（如最终合成）在依赖点停下问用户。
6. **skill 编辑**：见 §3.2。
7. **多 skill 冲突**：见 §3.3。
8. **工具划分与编排**：见 §4。
9. **权限**：见 §5。

## 7. 事件与接口协议〔逆向记录〕

- 后端 `service.flova.ai /api/v2`，对象存储 GCS，鉴权头 `x-ssid`（无 Authorization）。
- **提交/消费流解耦**：`POST /chat` 只回 `{stream_chat_id, agent_status}`；`POST /stream_chat/info {stream_chat_id, stream_type}` 拉 **NDJSON** 流，信封 `{id, type, data, timestamp, stream_chat_id}`，type ∈ {control, message, notification, ping, complete}；message.message_type ∈ {0, 8, 14} 承载富内容。
- `/chat` 字段含 `is_step_mode / is_paused / is_resume / resume_message_id / files / reference_resources`；message 带 `checkpoint_id`（resume 锚，可从任意消息恢复）、`credits_info`、`exploration_info`、`ref_resource_ids`。
- **媒体生成生命周期（notification）**：`media_prompt_draft_update`（含 model_type / tool_type / reference_media_items）→ `media_prompt_draft_generation_started` → `resource_update`（单资源就绪，前端只重取该资源）。
- **异步轮询**：`resource/task_status`（进度%）+ `resource/get` / `list_by_ids` / `asset_resources`。
- **上传三段式**：`resource/upload_ticket_v2`（sha256 秒传去重）→ PUT GCS → `resource/upload_complete`；上传即触发 FaceDetection / Transcode，每图产多模态分析卡。
- `project/get` 返回游标分页 message 流。

## 8. 积分与成本形态〔实测+逆向〕

- 逐动作扣费并在消息卡上带回执：生歌一次显示 `-5`（挂在 Media Assets 完成卡上）。
- 商品片全程账本〔逆向〕：起 2000 → 参考图 −22 → 视频 −315 → 音频首次 +10 → 时间线首次 +30 → 余 1673（净耗 ~327）。
- 逐消息 `credits_info` 随消息流下发。

## 9. 七阶段流程对照表（商品片实录〔逆向〕）

| # | 阶段 | 交互形态 | 卡点 | 产物 |
|---|---|---|---|---|
| 1 | Skill 加载 + 规格确立 | 徽章 + 参数表 | 否 | spec |
| 2 | 素材上传 + 多模态分析 | 三段式直传 + 每图分析卡 | 否（但**强制先上传实拍**） | 4× 图像分析（用途/构图/光线） |
| 3 | 故事板设计 | 左侧三段结构 | **是**（候选决策芯片） | Elements/Shots/Audio |
| 4 | 关键元素参考图 | 生成卡 + 进度% | **是** | 2× 元素图（Nano Banana 2） |
| 5 | 逐镜视频生成 | 注册→写提示词→并行生成 | **是** | 3× 镜头视频（Seedance 2.0） |
| 6 | 音频（VO+BGM） | TTS+BGM 并行 + 音色库检索 | 否 | VO（MiniMax）+ BGM |
