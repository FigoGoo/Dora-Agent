## 结论

字段级契约不在 Markdown 中重复维护。机器可校验文件是字段级事实源，Markdown 负责说明契约边界、成熟度、例外和缺口。

| 契约类型 | 字段级事实源 | 当前验证 |
| --- | --- | --- |
| 业务 RPC | `api/thrift/business_agent_service.thrift` | 18 个 service、43 个方法；所有方法使用单一 `req` 参数，并返回已声明 struct。 |
| 业务 HTTP API | `api/openapi/business-api.yaml` | 103 个 operation、246 个 schema；所有 operation 有 `operationId` 和基础错误响应；未发现 `additionalProperties: true` schema。 |
| Agent 工作台 API | `api/openapi/agent-workbench.yaml` | 12 个 operation、44 个 schema；覆盖 session、run、SSE、event replay、interrupt、snapshot。 |
| AG-UI 事件 | `api/agui/agent-workbench-events.schema.json` | 40 个 canonical event type 和 40 个 payload schema。 |

## 使用规则

- 字段、必填、类型、枚举、schema 名称和 request/response 以本索引列出的机器可校验文件为准。
- Markdown 契约正文如与机器可校验文件冲突，先以机器可校验文件作为当前实现事实，再回补 Markdown 的说明和成熟度状态。
- 新增或变更字段时，先更新 Thrift、OpenAPI、JSON Schema 或 fixture，再更新对应契约 Markdown 和 `docs/contracts/契约成熟度复核.md`。
- 不能把历史 `code-plan/**`、归档文档或口头约定作为字段级事实源。
