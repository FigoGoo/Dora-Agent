# 原子 Tool 清单定案 v1

> **⚠ 已并入终版** `2026-07-11-aigc-system-design-final.md`，本文件仅作过程存档，内容可能过期——以终版为准。

> 日期：2026-07-11　状态：清单封版（字段级规范另行细化）
> 本文档只定义**原子 Tool 层**；编排层（Agent 动态编排、注册面形态、卡点、事件）另行设计。

## 设计理念（先于一切细节）

1. **真正的 AIGC = 分析用户需求目的，然后动态编排。** 不预定义流程——固定流程就是 Dify（人画好工作流，LLM 只是节点）；Dora 里 LLM 是编排者，执行图是 Agent 每次按需求现场生成的产物。
2. **只有方向，没有场景。** 平台按四个模态方向（视频/图片/音乐/音频）组织能力，场景（故事片/MV/商品片…）是无穷且易变的，一律由 Skill 以领域经验（软知识，非硬流程）承载。
3. **原子 Tool 层是唯一固定物**，是编排的全部词汇表；四方向 ×（生成/编辑/装配/理解）的完备性 = 编排自由度的上限。能力进不进池由**方向完备性**决定；v1 实现与否只是工程排期，不做"场景需要吗"式取舍。
4. **用户 Tool = 编排产物实例**（Agent 动态生成的执行计划），不是预定义注册清单。用户感知这次编排的进度流，不关心内部工具如何运转。
5. **业务保障以切面绑定在原子工具上**（不绑图）：图形状随便变，调到工具，权限/扣费/入库/绑定必然发生。

## 通用契约

三要素定义（tool_key / tool_name / tool_description，description 面向编排层与开发者，原子工具永不直接面向最终用户）；统一执行契约 `run(ctx, inputs) → outputs | fail`（ctx 由 runtime 注入）；无状态；输入输出只传引用/结构化小对象，媒体字节永不过手；工具互不调用（数据由编排层路由）；不发事件（runtime 统一广播）；写操作幂等（同键重放返回同一结果）。

## 一、认知类（6 个）——调一次 ChatModel，产一个认知产物

| tool_key | tool_name | 什么情况下调用 | 边界 |
|---|---|---|---|
| `draft_spec` | 规格起草 | 新创作项目确立时；或用户要改创作方向、需要重立规格时 | 不落库、不确认；产出含"需用户裁决项"清单 |
| `draft_script` | 脚本创作 | 需要先产文本内容（故事/歌词/文案/解说词）再进行结构化创作时 | 不落库、不分镜 |
| `draft_storyboard` | 故事板起草 | 需要把创作意图落成结构化蓝图（元素/输出单元/音频层）时 | 不落库、不生成媒体 |
| `edit_document` | 文档修订 | 用户对既有规格/脚本/故事板提出修改时（修改意图涵盖翻译/改写类变换） | 只产 JSON Patch + 影响提示，不应用 |
| `write_media_prompt` | 提示词编写 | 任何生成动作派发前，为目标准备提示词时 | 模板按目标类型分发（可扩展，skill 可注入规则）；不选模型、不发起生成 |
| `understand_media` | 素材理解 | 用户上传或引用素材后，需要看懂才能规划/复用时 | 接口全模态（image/video/audio），v1 实现 image；只读，不产优化版 |

## 二、媒体执行类——按四个模态方向组织，每方向"生成 → 编辑/精修 → 装配"三段齐

共同边界：产 `resource_id`+摘要，**绝不**返回字节/base64/provider 原始负载；不扣费、不注册资产、不绑定（切面做）。Provider 适配器（即梦/Seedance/MiniMax…）在本层之下，是模型栈配置不是工具。

**图片方向**

| tool_key | tool_name | 什么情况下调用 | v1 |
|---|---|---|---|
| `text_to_image` | 文生图 | 无任何参考、从文字冷启动出图时 | ✅（image2 已有） |
| `image_to_image` | 图生图 | 有参考要保持一致性出图时（一致性锚点/素材优化版/单点重做） | ✅（image2 可覆盖） |
| `edit_image` | 图片局部编辑 | 对已有图"只改这一块"（蒙版重绘/消除/扩图）时 | ▢（v1 用 i2i 近似） |
| `compose_image` | 图文排版合成 | 海报/封面等需要图+文字层空间排版时 | ▢ |
| `super_resolution` | 超分增强 | 画质提升需求 | ▢ |

**视频方向**

| tool_key | tool_name | 什么情况下调用 | v1 |
|---|---|---|---|
| `image_to_video` | 图生视频 | 画面定格已确认、要让它动起来时 | ✅（seedance 已有） |
| `text_to_video` | 文生视频 | 无图直出视频时 | ▢ |
| `multimodal_to_video` | 多模态生视频 | 图+文+音组合驱动生成时；**参考驱动（运镜/风格参考视频）为入参能力**，不另设工具 | ▢ |
| `audio_driven_video` | 音频驱动视频 | 唇形同步（说话/演唱画面）时 | ▢ |
| `images_to_video` | 序列图生视频 | 多图序列成片段时 | ▢ |
| `video_interpolate` | 插帧 | 补帧/顺滑需求 | ▢ |
| `video_to_video` | 视频风格化/重绘 | 对已有视频做风格转换、重绘时（视频方向的"编辑段"，与图片方向 edit_image 对应） | ▢ |
| `extract_keyframes` | 关键帧提取 | 从已有视频取帧（理解/复用）时 | ▢ |

（"单元级重做出 variant"属重新生成机制，与 video_to_video 的"已有视频作输入的变换"是两回事——2026-07-11 对抗压测修正）

**音乐方向**

| tool_key | tool_name | 什么情况下调用 | v1 |
|---|---|---|---|
| `generate_music` | 音乐生成（器乐） | 需要按风格/情绪/时长产配乐时；**参考音频续写/扩展为入参能力** | ✅ |
| `generate_song` | 歌曲生成（歌词生歌） | 需要带人声的完整歌曲时 | ▢（能力池必备，实现排期待定） |

**音频方向**

| tool_key | tool_name | 什么情况下调用 | v1 |
|---|---|---|---|
| `text_to_speech` | 语音合成 | 文本要变成语音（旁白/台词/播报）时 | ✅（demo audio 占位；音色 v1 预设表） |
| `generate_sound_effect` | 音效生成 | 环境音/音效需求 | ▢ |
| `clone_voice` | 音色克隆 | 用户声音样本产出可复用音色资产时（音色资产为资产类型扩展，非媒体 resource） | ▢ |
| `separate_audio_tracks` | 音轨分离 | 已有音频拆人声/伴奏时 | ▢ |

**跨方向装配（时间线类）**

| tool_key | tool_name | 什么情况下调用 | v1 |
|---|---|---|---|
| `assemble_media` | 媒体装配 | 多资产按结构化时间线拼成品（排轨/对齐/混音/渲染）时；输出**视频或音频**参数化；字幕作为输入参数（由 script+TTS 时间戳确定性转换，不设工具）。能力定义须含：**clip in/out 裁剪点、叠加轨 overlay（logo/水印/画中画）**；单输入退化形态即可做格式转换 | ✅ |

## 三、数据类（3 个）——按数据域收拢，操作类型作参数

| tool_key | tool_name | 收拢范围 | 什么情况下调用 |
|---|---|---|---|
| `asset_store` | 资产落库 | 入库(传对象存储+resource 记录)/注册/加 variant/换 favorite/**绑定（独立绑定表，已拍板；绑定目标多态：文档结构节点/计划产出槽/会话交付物——任何编排形状的产物都有归属）**/软删（唯一删除入口，仅响应用户明确指令） | 生成产物就绪入库时（落库链必然调用）；用户选版本/换绑定/明确要删时 |
| `document_store` | 文档落库 | 定版(版本单调递增、结构 ID 定版生效)/应用 patch(base_version 乐观锁)/分析落盘(独立分析域只追加) | 认知草稿经确认要定版时；修改要应用时；分析要落盘时 |
| `dispatch_generation` | 任务派发 | 批量写 job 记录并入队（幂等键 = 计划实例 id+节点+目标+attempt；文档版本为可选组分——轻流程可无文档） | 目标与提示词就绪、把生成任务交给异步中心时 |

调用入口备注：数据类为双宿主库函数（图节点/异步 worker）；用户直接操作入口仅两个——手动编辑文档（apply patch 经数据接口）、前端删除按钮（软删经数据接口）。

建议默认（细化阶段可复议）：落库链"幂等重放向前"（每步幂等、断点重放、不回滚）；写权限按"入口×写域"白名单校验；写操作携带 actor 审计上下文。

## 四、护栏类（5 个）——确定性校验与资金；fail 是决策输入不是异常

| tool_key | tool_name | 什么情况下调用 |
|---|---|---|
| `precheck_inputs` | 前置校验 | 重节点执行前，校验**计划节点声明的输入引用是否真实就绪**（资产 ready/文档版本存在），缺则 fail-fast 返回缺口。不含业务顺序规则（"先规格后蓝图"等属 Skill 软知识+Agent 编排判断，工具层不固化流程） |
| `check_permission` | 权限校验 | 即将发生消耗性动作前（无副作用，与预扣分开） |
| `compliance_check` | 合规预检 | 提示词/文本即将送 provider 前；禁区必拦、拦截必过人、绝不静默改写绕过 |
| `reserve_credits` | 预扣费 | 消耗性动作确认执行、先锁定额度时（**v1 空实现占位**，已拍板留） |
| `settle_credits` | 结算回冲 | 生成成功结算/失败回冲时（v1 空实现占位） |

## 五、交互类（2 个）——人机卡点载体

| tool_key | tool_name | 什么情况下调用 |
|---|---|---|
| `request_confirmation` | 确认卡点 | 到达"用户不拍板不能继续"的位置：方向性决策、重消耗动作授权、合规/返工授权 |
| `request_user_input` | 收集卡点 | 需要用户给材料或开放信息：请上传素材、填偏好、补充意图 |

## 明确不设的（拍板记录）

- **检索类整个不设**（库内检索、外部检索）：v1 无需求；未来按新增大类处理。（2026-07-11 拍板）
- **进度/通知卡不设工具**：通知归 runtime 事件广播；卡点的定义就是"挂起等人"。

## 明确不是工具的（设施/机制）

异步生成中心（消费 job、并行、批次完成事件）；事件广播（runtime 统一发，节点无权私发）；Skill/Tool 中间件、意图识别（Agent 本体）；Provider 适配器（模型栈配置）；用户上传三段式（数据接口）；**编排域元操作**（编排库检索——v1 库目录注入 Agent 判断、执行计划合法性校验——runtime 提交时同步校验、沉淀参数化——编排层机制）。

> **层次原则：原子工具层 = 创作域词汇表。** 编排域的元操作（组计划/校验计划/沉淀计划）永不进入原子层，防止"编排编排自己"的层次污染。（2026-07-11 编排层总纲定案后的一致性审查确立）
