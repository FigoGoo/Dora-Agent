## 0. 设计理念

1. **真正的 AIGC = 分析用户需求目的，然后动态编排。** 不预定义流程——固定流程就是 Dify（人画好工作流，LLM 只是节点）；Dora 里 LLM 是编排者，执行图是 Agent 每次按需求现场生成的产物。
2. **只有方向，没有场景。** 按四个模态方向（视频/图片/音乐/音频）组织能力；场景无穷且易变，一律由 Skill 以领域经验（软知识，非硬流程锁）承载。场景化的东西只允许从沉淀里长出来，不允许初始预置。
3. **统一 node 库是唯一固定物**——编排的全部词汇表；原始 Tool（调模型直接交互）只是 node 的内核。四方向 ×（生成/编辑/装配/理解）的完备性 = 编排自由度上限。能力进池由方向完备性决定，v1 实现与否只是排期。
4. **GraphTool（用户 Tool）= 连线模板**：编排知识，不是执行体；执行体只有一个（计划调度器）。暴露面两层——路由面 = 场景模板目录，编排面 = node 词汇表；裸的原始 Tool 永不暴露。用户感知编排的进度流，不关心内部运转。
5. **业务保障是 node 的显式阶段**（不是隐形切面、不绑图）：预留/结算/补偿、落库/绑定全部画在 node 的阶段序列上——连线随便变，走到 node，保障必然发生且图上可读。
6. **层次原则**：原始工具层 = 创作域词汇表；编排域元操作（组计划/校验计划/沉淀计划）永不进入原子层。

## 1. 三层词汇表（原始 Tool → node → GraphTool）

> 2026-07-13 重排：本章取代原「原子层 36 工具」单层模型。原认知/媒体执行清单平移为 node 内核；原数据/护栏/交互三类溶解为 node 的显式阶段与卡点状态（去向表见 §1.2）。

### 1.1 原始 Tool = 调模型直接交互（唯一定义）

原始 Tool 收窄为**与模型直接交互的那一步**，是 node 的内核，永不单独暴露。通用契约沿用：三要素（tool_key / tool_name / tool_description，面向编排层与开发者）；`run(ctx, inputs) → outputs | fail`；无状态；只传引用/结构化小对象，媒体字节永不过手（**provider_ref 仅 worker 进程内流转**，正式 `resource_id` 由 node 落库阶段物化；绝不向上返回字节/base64/provider 原始负载）；互不调用；不发事件；写幂等。Provider 适配器在本层之下（模型栈配置，非工具）。

内核三族：

**认知族（调一次 ChatModel，产一个认知产物）**

| tool_key | tool_name | 什么情况下调用 | 边界 |
|---|---|---|---|
| `draft_spec` | 规格起草 | 新创作项目确立/改方向重立规格 | 不落库不确认；产出含"需用户裁决项"清单 |
| `draft_script` | 脚本创作 | 需先产文本内容（故事/歌词/文案/解说词） | 不落库不分镜 |
| `draft_storyboard` | 故事板起草 | 把创作意图落成结构化蓝图（元素/输出单元/音频层） | 不落库不生成媒体 |
| `edit_document` | 文档修订 | 对既有文档提修改（含翻译/改写类变换） | 只产 JSON Patch+影响提示，不应用 |
| `write_media_prompt` | 提示词编写 | 任何生成派发前为目标备提示词 | 模板按目标类型分发（skill 可注入规则）；不选模型不发起生成 |
| `understand_media` | 素材理解 | 素材需看懂才能规划复用 | 接口全模态，v1 实现 image；只读不产优化版 |

**生成族（调一次生成模型）**——四模态方向组织，✅ v1 实现 / ▢ 留格子沿用原判定：

| 方向 | v1 实现（✅） | 留格子（▢） |
|---|---|---|
| 图片 | `text_to_image`（image2 已有）、`image_to_image`（image2 可覆盖） | `edit_image`（v1 以 i2i 近似）、`compose_image`（图文排版）、`super_resolution` |
| 视频 | `image_to_video`（seedance 已有） | `text_to_video`、`multimodal_to_video`、`audio_driven_video`、`images_to_video`、`video_interpolate`、`video_to_video`（视频的编辑段） |
| 音乐 | `generate_music`（器乐；参考续写为入参能力） | `generate_song`（能力池必备，排期待定） |
| 音频 | `text_to_speech`（demo 占位；音色 v1 预设表；**输出含分段/句级时间戳**） | `generate_sound_effect`、`clone_voice`（音色资产）、`separate_audio_tracks` |

**执行族（媒体处理执行器，不调模型）**：`assemble_media`（✅ 时间线类：排轨/对齐/混音/渲染；输出视频或音频参数化；须含 clip in/out、叠加轨 overlay；字幕作输入参数；单输入退化形态即格式转换；空间排版归 compose_image）、`extract_keyframes`（▢）。这是对"原始 Tool = 调模型"的唯一放宽：内核也可以是确定性媒体处理执行器。

### 1.2 Node = 原始 Tool 的封装（内核 + 显式阶段 + 状态机）

node 是编排与执行的**统一复用单元**（梁 1：同一个"分析 node"在图片场景、故事板场景、上传自动分析里是同一实现）。

**三种内核形态 + 卡点（阶段画在图上，不做隐形切面）**。前三种有内核（调模型/执行）；**卡点无内核——它就是 node 的「等人」状态本身**（2026-07-14 定案：卡点由 node 状态决定，不是第四种内核），计划里由卡点声明落成一个纯状态步：

| node 形态 | 内核 | 阶段序列 | 挂起 | 扣费 |
|---|---|---|---|---|
| 认知 node | 认知族 | 取输入 → [预留] → 调 ChatModel（坏输出自动重试 ≤2）→ 产物落盘 → [结算] → 发事件 | 无（同步） | 声明了预留/结算才扣 |
| 生成 node | 生成族 | 取输入 → 预留 → 派发批次（=一批 job）→〈挂起·等批次〉→ 批次终结唤醒 → 落库+绑定+结算 → 发事件 | 等批次 | 必扣 |
| 执行 node | 执行族 | 取输入 → 执行 →〈可能挂起〉→ 落库+绑定+[结算] → 发事件 | 视情况 | 视处理量 |
| 卡点（状态） | 无内核·纯「等人」状态 | 呈现（确认卡/收集卡）→〈挂起·等人〉→ resume → [定版/绑定] | 等人 | 否 |

- **状态机（每个 node 一份）**：`待启动 → 运行中(含重试) → [挂起(等批次｜等人｜等审)] → 成功 | 失败 | 取消`。
- **扣费三条腿**：预留（前）、结算（成功后）、**补偿退回**（失败/取消，走 billing.compensation 事件链）。哪些 node 扣费 = 哪些 node 声明了预留/结算阶段——图上直接可读；生成 node 必扣，认知 node（如分析）按配置。
- **生成 node ↔ 批次**：一次生成 node = 一批 job（`n` 参数 = 同目标备选变体）；批次按 SuccessPolicy 判整体成败（允许部分失败）、**批次终结才唤醒**。多目标（多镜头）不是 `n`，是计划层多个独立 step。
- **状态聚合**：node 挂起(等批次) → 计划 `waiting_jobs`；挂起(等人/等审) → `waiting_user`；运行/待启动 → `waiting_agent`。node 状态向上聚合即 §3.6 计划状态机，不另发明状态。

**原数据/护栏/交互三类的去向**（不再是独立工具，约束原文继续生效）：

| 原工具 | 去向 |
|---|---|
| `asset_store` / `document_store` | node 后置阶段（产物落盘/落库+绑定：绑定独立表、软删唯一入口、重生成永不覆盖、版本乐观锁、分析域只追加）；卡点 resume 的定版 |
| `dispatch_generation` | 生成 node 的派发阶段（幂等键 = 计划实例+节点+目标+**派发序号 attempt**；预注册 generating 资产；重试预算与用户调整分离——全部沿用） |
| `precheck_inputs` | node 取输入阶段的就绪校验（不含业务顺序规则，那属 Skill+Agent） |
| `check_permission` | node 前置阶段（"入口×写域"白名单仍在计划校验时物理强制） |
| `reserve_credits` / `settle_credits` | 预留/结算/补偿三阶段（v1 空实现占位不变） |
| `compliance_check` | **不设独立出口审**：①入口一道——用户素材/原话进系统时校验（可异步 →〈等审〉挂起）；②生成时 provider 拒绝 = node Failure（`Retryable` 区分限流重试与内容硬拒），计划层处置沿 R3（拦→⏸ 告知调整方向，硬性禁区直接终止）；③交付标识/留痕归 §5 |
| `request_confirmation` / `request_user_input` | 卡点两呈现形态（确认卡/收集卡；状态驱动，非内核） |

明确不设与非工具照旧：检索类不设；进度通知归事件广播；异步生成中心/事件广播/中间件/意图识别/Provider 适配器/上传三段式/编排域元操作皆为设施非词汇。

**合计：内核 v1 实现 12（认知 6 + 生成 5 + 执行 1），留格子 14，内核总盘子 26；内核形态 3 + 卡点（状态，2 呈现形态）；场景模板 = 4 模态直呼 + 故事板（复杂路径规划段，门控进入，见 §2）。**（原 36 盘中的数据 3/护栏 5/交互 2 溶解为阶段与卡点状态，不再计为内核。17 路径对抗压测教训继续在案：结论自洽 ≠ 路径可达。）

### 1.3 GraphTool = 连线模板（编排知识，不是执行体）

- **定性**：GraphTool 是「node 怎么连」的**模板知识**。Agent 路由到它后按用户语意实例化——绑参数、消解事实分支、批量展开——产出**纯 DAG 计划（节点 = node）**，交唯一的计划调度器执行。**不存在图执行引擎**（mediagraph 固定图路线已作废且为验收翻车真凶，不复活）。
- **薄模板尺子（梁 2）**：模板内条件只允许「意图不变、仅由输入事实路由」的分支（有/无素材 → 图生图/文生图），作为**实例化注记**在实例化时消解；凡靠对话/反馈/数量/跨模态才能决定的，一律上抬 Agent（计划层组合或评估点续编），不进模板。
- **模板不改内部连线**：实例化 ≠ 改线。语意超出模板 → 计划层组合（§2.4 档 2）或兜底现场连线（档 3），模板本身原样跑——固化即质量稳。

## 2. 场景模板与暴露面（GraphTool 库）

### 2.1 暴露面（两层，2026-07-13 定案）

| 层 | Agent 看到什么 | 用途 |
|---|---|---|
| **路由面** | **4 个模态模板**（用户命中/@直呼）：图片生成 / 视频生成 / 音乐生成 / 音频生成 | 判断进哪条流程；模板的**输入要素清单**同时是引导话术数据源（语意不全 → 按清单主动引导收集，把用户往流程上引导） |
| **场景 skill（匹配面）** | **注册的复杂场景题材经验包**（MV / 商品片 / 剧情… 可扩展，有命中描述）：Agent 按用户需求匹配命中 | 只与复杂场景（故事板）对齐——命中即走故事板并把题材经验注入编排各决策点；简单模态不匹配 skill。新增场景=注册新 skill（不改骨架/流程图） |
| **编排面** | **node 词汇表**（认知/生成/执行 内核 + 卡点状态，三要素+入参出参概要） | 计划层组合与兜底连线的原料 |
| 不暴露 | 原始 Tool（裸调模型/provider 细节） | 封装于 node 内核 |

使用双通道照旧：①需求命中（M1 分析 → M2 路由）②用户 **@ 直呼** = 路由面强制命中（跳过命中判定，实例化仍经 Agent：深浅裁决、参数补齐、缺参走收集卡）。加工类（改图/给视频加字幕配乐等）**不单设路由项**：由各模态模板的实例化注记按「带来物」消解（有素材即走加工分支；原 image_edit/video_edit 模板知识并入对应模态模板）。**初始库 = 分类级（模态方向级，平台中性）；场景化模板只允许从沉淀长出（§2.4 档 4）。**

**入口顺序（2026-07-14 二次校准）：先经 M2 匹配场景 skill——命中则按 skill 描述的协作方式走**（skill 会写明协作流程，可指定「走故事板链」并做题材化阶段编排，如 MV 驱动音频前置）；**复杂度门是「未命中 skill」时的默认判定，不是 skill 之前的前置门**。命中 skill 后不再过复杂度门（skill 说了算）。

**复杂度门（2026-07-14 定案，融合原三个独立问题为一件事；作为未命中 skill 的默认判定）**：M1 从**用户规格的关键信息**判断——单产物需求（一句话/一张图/一段视频/一首曲/一段旁白）→ 直接路由到对应模态模板（档 1，主通道）；复杂项目（多镜头叙事/一致性系列/需分镜规划）→ Agent 组合编排（档 2，通用故事板骨架、无题材特化）。**故事板不作为用户直呼的路由项**——它是复杂路径的规划段，由命中 skill 指定或由本门兜底进入（§2.3）；多镜头成片、一致性系列都是复杂侧的组合产物，同样不进模板。门判不准 → 靠 M1 反问边界兜（§3.2），不设独立卡点。**门是「收窄非废除」**：命中 skill 时由 skill 回答「是否复杂」，门只在未命中时粗判兜底；且 skill 库初期小甚至空，**初期门是主力、skill 命中是少数**，故通用故事板骨架必须独立好用（§6.10）。

**skill「协作方式」的粒度（防回潮）**：命中 skill 携带的「协作方式」= **「引用哪副协作骨架（故事板模板 / 沉淀模板）+ 题材化阶段编排参数」，不是 skill 自己从零定义 node 连线**。骨架实体永远在模板层，skill 只做「选骨架 + 题材化排布 + 注经验」——skill 与模板界线清晰、不重叠（详见 §6.10）。

**M1 需求分析框架 = 需求→产物矩阵**：行=用户带来物（只有想法/文本/图/音频/视频/平台产物），列=想拿走的产物方向。分析产物四要素：产出方向、带来物、编排深浅、约束偏好——直接充当计划元信息与 M2 命中输入。

### 2.3 故事板场景 = 模板骨架 + Agent 编排（Skill 可选注入题材经验；定案，梁 3 修正见 §6.9#7）

「故事板 → 完整视频」**不是一张固定长图**，是 Agent 跑的一张执行计划：

```
先匹配场景 skill：命中 → 按 skill 协作方式走（skill 可指定「走故事板链」+ 题材化编排）
未命中 → 复杂度门（M1 从用户规格关键信息判）兜底：单产物 → 直调模态模板，不启动故事板
命中 skill（指定故事板）或未命中判复杂 → 故事板模板（规划段，门控/skill 进入非直呼）→ 逐段生产（Agent 计划层组合）：
  锚点图段 = N×图片生成模板 → ⏸锚点确认
  → 镜头段 = N×视频生成模板（每镜头独立 step，可单独重做）
  ∥ 音频段 = 音乐/音频模板
  → 装配node（尾段执行 node，不独立成路由项）→ 交付
```

- **骨架在故事板模板里**（编排知识：阶段序列、依赖、默认卡点位置——§1.3），Agent 逐段实例化并填血肉（内容/参数/串并行/异常）。**场景 skill 是题材经验层**（软知识：某题材下的素材分析侧重/规划方式/模型偏好/暂停节奏/领域规则），**像 GraphTool 一样注册到 Agent、按用户需求匹配命中**（§2.1 匹配面），命中后注入题材经验；**只与复杂场景（故事板）对齐，简单模态不用 skill**。同骨架 + 不同 skill = 不同题材成片；未命中走通用编排（能跑但泛）。**新增场景=注册新 skill，不改骨架**。完整走查与注入点见详设 §3.1。
- 必须覆盖（Flova 实测查漏所得，机制均已有着落）：复杂度门；每镜头独立可寻址（单镜头重做不重跑流程、不重置时间线）；中途改需求 → 影响评估 → 复用已确认产物 → 活计划修订（§3.6②）；单点失败不卡整条（SuccessPolicy+批次终结）；卡点上操作全开放（精改/跳过/回退/部分重做）；已确认产物保护（§5 favorite 锁定）。

### 2.4 动态的度（四档，2026-07-13 定案）

| 档 | 什么情况 | 谁在动 | 质量来源 |
|---|---|---|---|
| **1 命中固化流程** | 常见场景 | 只绑参数、消解事实分支、×N 展开——**不改模板内部连线** | 验证过的线原样跑（**主通道**） |
| **2 计划层组合** | 语意超出单个模板 | Agent 把模板实例+node 串成链（如 翻译node→图片模板；故事板成片长链 §2.3） | 链上每段仍是固化件 |
| **3 兜底现场连线** | 组合也盖不住 | Agent 从 node 词汇表连新图 | 计划校验器五查兜底（§3.4） |
| **4 沉淀** | 兜底跑成且有复用价值 | 新连线入库成固化流程（用户确认才入库，M6） | 库越用越厚，兜底越走越少 |

编不出 = 诚实告知（合法终点）。两个极端被结构性排除：没有档 3 = Dify；没有档 1 = 质量不稳又慢又贵。**改已固化模板的内部连线不属于任何一档**（质量方差爆炸/不可测试/架空沉淀闭环，禁止）。

### 2.5 通用执行规则 R1–R7（原六图共享；2026-07-13 重排后读法）

> **R1 废除**——隐形切面模型被 §1.2 显式阶段取代（保障语义不变，形态从"织入"改为"画在 node 阶段上"）。**R2 收敛为生成 node 的执行语义**（§1.2 阶段序列即其 node 化表述）。**R3/R4 属计划层**（评估点/卡点处置，继续生效）。**R5 = 认知 node 语义、R6/R7 不变，继续生效。** 原文保留如下：

- **R1 切面**：媒体执行节点自动织入——前置 `check_permission → compliance_check → reserve_credits`；后置（worker 侧落库链）`upload_to_storage(物化 resource) → asset_store(置 ready/挂 variant) → bind_asset → settle_credits(失败回冲)`，每步幂等、断点重放向前。**切面与数据/护栏工具只由 runtime 织入，不进 Agent 编排词汇表、不可作计划节点**——杜绝双重织入。
- **R2 派发-挂起-唤醒**（一切媒体执行的形态）：`dispatch_generation`（批量 job + 预注册 generating 资产）→ 计划 suspended(waiting_jobs)、checkpoint、本轮 run 收尾 → 异步中心逐 job：worker→provider→落库链→逐个广播事件（前端逐个点亮）→ 批次终结 → wakeup → 计划恢复。**活性保障**：job 带执行超时（超时置 failed 计入批次终结）；批次看门狗周期核对 suspended(waiting_jobs) 计划的批次真实状态并补发唤醒（wakeup 崩溃丢失的兜底）；唤醒幂等。
- **R3 评估点 ◆ = 回 Agent 续编点**，两型：后置质量型（全成→续段｜部分失败→补派发 attempt+1，上限 2｜达上限→带缺口进 ⏸｜provider 合规拦→进 ⏸ 告知原因与调整方向，经用户授权改向后重写重试；硬性禁区不适用授权，直接终止该目标）；前置可行性型（可行→续｜歧义→request_user_input 澄清｜不可行→说明+替代，计划 cancelled）。
- **R4 卡点 ⏸ 三出口**：确认→续段｜调整第 k 项(带反馈)→重写重派→回本卡点｜中途改需求→影响评估→授权→活计划修订（已执行不回滚，被裁置 skipped）。
- **R5 认知类同步执行**（只有媒体执行经 R2）；失败语义：ChatModel 调用失败/坏输出自动重试 ≤2，耗尽 → 计划 failed 并告知原因；compliance 软拦 flag → 生成前过 ⏸，硬拦直接拒绝。
- **R6 深浅裁决与升级**：深浅由 Agent 定；浅结果不满足可原地升级，已产资产复用。
- **R7 卡点开关**：深编排 ⏸ 默认开，用户可声明关闭后续某类确认；**边界：合规 ⏸ 与涉用户素材 ⏸ 不可关**（权限性质非效率性质）。

## 3. 动态编排

### 3.1 主流程（方向性总纲）

```
用户需求 → Agent 分析需求目的（M1）
              ↓
       检索编排库（M2）——动态四档见 §2.4
        ├─ 命中 → 模板实例化（绑参/消解注记/×N 展开，不改内部连线）→ 执行计划
        ├─ 语意超出单模板 → 计划层组合（模板实例+node 串链）→ 执行计划
        └─ 全不命中 → 兜底：从 node 词汇表现场连线（M3）→ 执行计划
              ↓
       同一个 runtime 执行（不区分计划来源）
              ↓
       成功且有复用价值 → 沉淀回库（M6）
              ↑ 库越用越厚，永远有动态编排兜底（不是 Dify 的根本原因）
```

### 3.2 M1 需求分析（Agent 本体）

产物四要素（产出方向/带来物/深浅/约束偏好，见 §2.1 矩阵）。反问边界：**必问**——定位不了矩阵格子、提到素材没给；**不问用默认**——能被后续卡点兜住的细节偏好（原则：反问有打断成本，卡点能兜的不提前问）；完全无方向→引导话术介绍四方向能力。**引导数据源 = 场景模板的输入要素清单**（§2.2）：语意不全时按清单主动收集，把用户往流程上引导。**复杂度门是「未命中 skill」的默认判定**（不是 skill 之前的前置门）：先经 M2 匹配场景 skill，命中则按 skill 协作方式走（skill 可指定进故事板）；未命中才由本门从用户规格关键信息（是否多镜头/叙事/一致性系列/时长结构）判单产物 vs 复杂项目——前者路由模态模板，后者进通用故事板编排（§2.1）；判不准走上述反问边界，不设独立卡点。

### 3.3 M2 库命中

v1 库小：目录整体注入上下文，Agent 直接判（不建检索服务）。三档处置：明显命中→直取绑参并说明｜模糊两可→出候选让用户选（不替用户猜方向性决策）｜不命中→兜底动态编排。**"永不拒绝"指路由层永远有 M3 兜底（不存在"库里没有"式拒绝）；M3 编排本身可能失败并诚实告知（§3.4），失败是合法终点。** @ 直呼旁路跳过 M1/M2 的命中判定。

### 3.4 M3 动态编排机制

- **词汇表注入**：**编排面 = node 词汇表**（认知/生成/执行 内核 + 卡点状态，三要素+入参出参概要）**+ 路由面 = 4 模态模板目录（故事板门控进入，不注入为直呼项）** + 激活 skill + M1 四要素。数据/护栏已内化为 node 显式阶段（§1.2 去向表），**不是词汇、不可作计划节点**；留格子不注入（格子实现自动进入）。复用 turncontext 瞬态注入机制。
- **计划生成**：产出 8 要素计划（结构化输出）；**一切执行皆计划实例**（单节点也是），执行面零分叉；"提交计划"是编排域元能力（不进原子层）。
- **合法性校验**（runtime 提交时同步，五查）：结构（无环/node 存在且已实现/边指向真实节点）、参数（必填绑定/类型匹配/槽已实例化）、权限（入口×写域白名单）、保障（node 阶段声明完整：扣费/落库/绑定阶段齐备可执行）、预算（v1 以**生成 job 数阈值**计——credits 空实现期无金额可算，接计费后升级为金额；超阈值不拒绝→转预览 ⏸）。
- **修复循环**：结构化错误回 Agent → 修计划重提 → 上限 2 次 → 仍败则诚实告知编排不出+原因，不硬跑。
- **计划预览 ⏸**：v1 触发仅"深编排首提"（预算阈值因计费空实现暂无数据，降级为日志观测，计费落地后启用）→ 先给用户看"打算怎么做、预计消耗"；轻计划直接跑。

### 3.5 M4 执行计划（8 要素）

元信息（来源/需求摘要/方向）｜节点集（**node 引用**+参数绑定：字面值/上游引用/参数槽）｜依赖边（纯 DAG）｜批量展开｜卡点声明（问用户）｜评估点声明（回 Agent）｜预算预估｜成功判据。
**分支兑现（定案）：计划内不放 if/else，分支 = 评估点 + Agent 续编（分段编排）。** 理由：判断留给 LLM 才是"LLM 是编排者"的彻底版；计划保持纯 DAG；已知代价是多几次 LLM 调用，换判断质量与架构简单。

### 3.6 M5 runtime 执行：计划实例状态机

```
draft（Agent 编排中，未提交）→ running ⇄ suspended(waiting_user/agent/jobs)
draft / running / suspended → cancelled（用户拒绝或终止）
running → succeeded / partial_succeeded / failed
节点/展开实例：pending → running → succeeded / failed(重试耗尽) / skipped(修订被裁)
```
**计划预览 = suspended(waiting_user)**（受挂起统一模型管辖，checkpoint/唤醒同一套）；draft 仅指未提交。
① 挂起统一：一个 suspended + 三原因（waiting_user/waiting_agent/waiting_jobs），checkpoint/唤醒/展示一套逻辑。
② 活计划：续编=追加；修订=改未执行（裁掉置 skipped）；已执行永不回滚；终态=无未执行且无挂起。
③ 挂起即释放：不占运行时资源，当轮 run 正常收尾；三种唤醒对号（用户 action/Agent 续编/job 批次 wakeup——复用现有 wakeup）。
checkpoint/resume：挂起必 checkpoint（计划+节点状态+产物引用+文档版本快照）；resume 幂等一次性；恢复时版本冲突检测（变了不硬续，转确认）。
**终态映射**：succeeded=全部节点成功｜partial_succeeded=quorum 通过但存在 skipped/缺口（用户选择跳过后交付）｜failed=M3 修复耗尽/认知类重试耗尽/批次全败且用户放弃｜cancelled=预览拒绝/前置不可行终止/用户主动终止。
**并发约束（v1）**：同一会话同一时间**一个活动计划**（running/suspended）；活动计划挂起期间的新需求由 Agent 判断并入当前计划（续编）或排队；多计划并发调度列入 v2。

### 3.7 M6 沉淀（v1 简化）

动态编排成功后 Agent 标记"可复用候选"，**用户确认才入库**（不自动沉淀）；参数化机制后置。**v1 沉淀 owner = 人工**：把跑通的编排写成模板数据，归"模板数据化"任务，自动化机制 v2。

## 4. 异步任务（异步生成中心）

- **定位**：共享基础设施（非工具非节点）；一切媒体生成的唯一执行环境（生成族内核唯一入口 = worker）。**生成 node 的「派发→挂起·等批次→批次终结→唤醒」阶段（§1.2）即由本中心承接**——原 R2 收敛为生成 node 的执行语义。
- **执行形态**：R2 循环（见 §2.5）。job 幂等键 = 计划实例+节点+目标+attempt。
- **job 状态机**：queued → running → succeeded / failed / cancelled（既有代码）+ **执行超时**：running 超时置 failed 计入批次终结——活性保障，新增；回调按 job_id + status_version 幂等。
- **批次语义**：逐 job 完成即广播事件（前端逐个点亮）；**批次终结**（全部 job 终态）才 wakeup 计划恢复——批次结果交 ◆ 按 R3 处置（含 quorum 部分成功）。**批次看门狗**：周期扫描 suspended(waiting_jobs) 计划、核对批次真实状态、补发丢失的唤醒（服务崩溃窗口的兜底）；唤醒幂等。
- **与计划状态机衔接**：派发即计划 suspended(waiting_jobs)；job 寿命可跨 run；唤醒复用现有 wakeup 机制（会话级事件流，不绑 run）。
- 现有代码基础：`generation/` Dispatcher→Redis Queue→Worker（并发 4）+ wakeup——存量最成熟，delta 为批次终结判定与计划挂起对接。

## 5. 资产生成和绑定

- **三层模型**：文档结构（element/unit/audio_layer）→ asset（逻辑资产：variants[]/favorite/source）→ resource（物理文件@对象存储）。
- **绑定独立表**（拍板）：故事板文档只存创作结构（人改的），绑定/生成状态是系统事实（机器写的）——写路径分离不撞锁；前端状态树投影时合成两域。
- **交付物模型 Deliverable（2026-07-12 裁决升级为一等设计物）**：一切生成产物绑定到 target-中立的交付目标，v1 两态：`storyboard_slot`（故事板槽位，走版本冲突检测+候选审批链）| `session_deliverable`（会话交付物：无版本冲突检测、无候选审批，直接交付+事后 variant 重做）。**落地前置（第 0 步）**：现实现的 BindingToken/候选审批/Barrier/Finalization 均硬锚 StoryboardID，须贯穿性改为可选——非加性改造，动持久底座保护面。
- **结构变更与在途生成的协调**：文档改版期间在途 job 照常完成；绑定时若目标结构 ID 已不存在 → 产物归入**孤立产物区**并进 ⏸ 告知用户处置（重新挂载/丢弃）；计划恢复时版本冲突检测照旧。
- **落库链**（生成/执行 node 的后置阶段，worker 侧）：upload_to_storage(物化 resource) → asset_store(置 ready/挂 variant) → bind_asset → settle_credits；**幂等重放向前**（每步幂等/断点重放继续完成/绝不回滚，孤儿 resource 靠 v2 对账）。
- **交付标识与留痕**：生成产物的内容标识（水印/标识）与违规处置留痕是**交付物属性**，归本层落地；不进生成 node 阶段（送审定案见 §1.2 去向表）。
- **资产生命周期**：**派发时预注册**（registered/generating，前端占位）→ ready →（用户确认）confirmed；重生成/调整 = 并列 variant（新派发序号），**favorite 不自动切换**——变更 favorite 是用户显式动作，已确认资产的 favorite 锁定（"未经授权不得替换"的物理兑现）；用户上传原件永久只读；删除唯一入口 = 软删 + 用户明确指令。
- **权限物理化**：写权限按"入口×写域"白名单在计划校验时强制（skill 改不了、Agent 绕不过的物理层）。
- **审计**：一切写操作携带 actor 上下文（session/run/计划实例/节点/job_id）。

### 6.10 skill 机制校准（2026-07-14，用户纠正「剥离过头」）

§6.9#7 去污染时把 skill 从故事板几乎剥光、并写「不预置任何题材 skill」——矫枉过正，因为**故事板恰是多场景 skill 的主战场**。校准：

- **场景 skill = 注册到 Agent、像 GraphTool 一样按用户需求匹配命中的「复杂场景题材经验包」**（MV/商品片/剧情…），有命中描述；**只与故事板（复杂场景）对齐**，简单模态不匹配 skill。
- 命中后把题材经验注入故事板编排各决策点（素材分析侧重/规划/模型偏好/暂停节奏/领域规则）；**新增场景 = 注册新 skill，不改骨架、不改流程图**。
- **不变的部分**：skill 仍是题材经验层、不是骨架（骨架在模板，§6.9#7 这点仍成立）。
- **澄清**：「从沉淀长出、不预置」针对的是**场景化模板**（连线组合），**不是 skill**——各场景本就该有各自题材经验 skill（§0 理念 2 本意）。
- **入口顺序（同日二次校准）**：**先匹配 skill → 命中即按 skill 描述的协作方式走**（skill 携带「协作方式」，会指定走哪条链 + 阶段序列题材化编排，例：「规格→故事板→元素图→镜头→音频→剪辑」）；**复杂度门退为「未命中 skill」的默认判定**，不再是进故事板的前置门。命中 skill 后不再过复杂度门。这与动态四档「命中优先、兜底现场判」（§2.4）一致。
- **复杂度门是「收窄」不是「废除」**：原门被塞了两件事——①判是否复杂场景 ②单产物路由到哪个模态；①在命中 skill 时由 skill 回答（更准）、未命中时才由门粗判，②本就是 M1/M2 路由。门的真身 = **「未命中 skill 时，粗判单产物→模态 / 复杂→通用故事板骨架」的兜底分类器**，归位而非作废。
- **初期诚实边界**：skill 库遵循「初始=分类级、场景化从沉淀长出」，**初期很小甚至空**——故初期绝大多数复杂需求会「未命中 skill → 走门 → 通用故事板骨架」，**门才是主力、skill 命中是少数**。这要求**通用故事板骨架必须独立好用、不依赖 skill 才能跑复杂项目**（未命中走通用编排「能跑但泛」已保证此点）。
- **skill「协作方式」的粒度（堵回潮）**：skill 携带的「协作方式」= **「引用哪副协作骨架（故事板模板 / 某沉淀模板）+ 题材化阶段编排参数」**，**不是 skill 自己从零定义 node 连线**。骨架实体永远在模板层，skill 只做「选骨架 + 题材化排布 + 注经验」——由此 skill 与模板界线清晰、不重叠，也堵住「skill 又变回骨架」的老毛病（本轮已在此线反复两次，见 §6.9#7 与本节）。
