# AIGC 编排层设计

> **⚠ 已并入终版** `2026-07-11-aigc-system-design-final.md`，本文件仅作过程存档，内容可能过期——以终版为准。

> 日期：2026-07-11　状态：**方向性总流程已定案**，模块逐个细化中
> 前置：原子 Tool 层已封版（见 2026-07-11-aigc-atomic-tools-v1.md），本层在其之上
> 理念：真正的 AIGC = 分析用户需求目的后做编排；不固定用户思维（固定即 Dify）；只有模态方向没有具体场景

## 总流程（方向性，一切细节的框架）

```
用户需求 → Agent 分析需求目的
              ↓
       检索编排库（已沉淀的流程模板）
        ├─ 命中 → 取出模板，绑定本次参数（素材/规格/偏好）→ 执行计划
        └─ 未命中 → LLM 从原子工具词汇表现场编排 → 执行计划
              ↓
       同一个 runtime 执行（状态机/并行/卡点/切面/事件——不区分计划来源）
              ↓
       执行成功且有复用价值 → 沉淀回编排库（参数化成模板）
                                    ↑ 库越用越厚，但永远有动态编排兜底
```

三个关键性质：
1. **执行计划同构**：库模板与动态编排产出同一种数据结构，runtime 只认这一种输入，执行面零分叉。
2. **库是沉淀物不是预画物**：开放生长 + 动态编排兜底，用户思维不被固定——与 Dify 的本质区别。
3. **"用户 Tool"的真身** = 库里沉淀的编排模板（含用户主动 @ 的预工具）。

## 模块地图（待逐个细化）

| # | 模块 | 一句话职责 | 主要待定问题 |
|---|---|---|---|
| M1 | 需求分析 | Agent 分析用户目的：产出方向（视频/图片/音乐/音频）、交付物、素材、约束、复杂度 | 分析产物的形态；何时反问用户补充信息 |
| M2 | 库检索与命中 | 拿需求分析结果匹配编排库 | 命中判定机制（谁判、依据什么）；模糊时的处置（直选/出候选/兜底动态编排） |
| M3 | 动态编排 | LLM 从原子工具词汇表现场组执行计划 | 词汇表如何注入；产出计划的合法性校验（白名单/无环/参数类型）；校验失败的修复循环 |
| M4 | 执行计划（载体） | 库模板与动态编排的同构产物，runtime 的唯一输入 | **构成要素已定案（见下节）**；字段级数据结构待细化 |
| M5 | runtime 执行 | 状态机推进、并行、卡点、切面织入、事件广播、checkpoint/resume | 大部分可继承既有底稿（07-10 稿 §3 节点契约/状态机/调度/失败语义），需按"动态图"改造 |
| M6 | 沉淀 | 成功编排参数化成模板入库 | 入库条件（谁判断有复用价值、要不要用户确认）；参数化规则；检索元数据 |

Skill 定位（贯穿）：领域经验软知识——M1 参考它理解需求、M3 参考它编排与写提示词；不是硬流程锁。

## M4 执行计划：构成要素（2026-07-11 定案·概念级）

| 要素 | 内容 |
|---|---|
| 1. 元信息 | 来源（库模板 id / 动态编排）、需求摘要、产出方向 |
| 2. 节点集 | 每节点：原子工具引用 + 参数绑定（字面值 / 上游输出引用 / **参数槽**——模板复用的关键） |
| 3. 依赖边 | 纯 DAG，无环——串并行由依赖客观决定 |
| 4. 批量展开 | 哪个节点按目标集展开 N 实例并行 |
| 5. 卡点声明 | 何处停、候选 actions——**问用户**的停点 |
| 6. 评估点声明 | 何处停、**回 Agent**的停点 |
| 7. 预算预估 | 预计消耗、重动作清单——执行前给用户预期 |
| 8. 成功判据 | 完成定义、部分失败算不算成 |

**分支的兑现方式（定案）：计划内不放 if/else，分支 = 评估点 + Agent 续编（分段编排）。**
运行时才能知道结果的判断（生成质量、素材可用性）不预写成条件边，而是声明评估点：执行到此挂起、结果回 Agent、Agent 看实际结果续编下一段。
理由：①判断留给 LLM 才是"LLM 是编排者"的彻底版，if 进计划数据结构等于把编排逻辑固化进数据（半个 Dify）；②计划保持纯 DAG，runtime 是纯状态机，不发明条件表达式语言；③编排时已知的分支（有没有素材）直接编对应路径，无需 if。
已知代价：分支密集的需求变成"执行一段→回 Agent→续编一段"的多轮节奏，多几次 LLM 调用——用调用成本换判断质量与架构简单。
草图五特性中的"逻辑判断（分支）"由本条兑现。

## 初始编排模板集（用户 Tool v1，2026-07-11 定案方向）

**定位**：编排好的产物——编排库的冷启动内容，基于原子工具词汇表编排而成，用户可直接使用。与原子工具同样有三要素（tool_key / tool_name / tool_description，description 即 M2 命中判定的匹配依据）。**不是 Agent 注册面**：是库存的编排产物，可被命中、可被沉淀迭代、可被用户直呼。

**使用通道**：①需求命中——Agent 分析需求后从库取模板绑参执行；②**@ 直呼**——用户在输入框直接点名模板，跳过 Agent 编排决策，参数不全走收集卡点补齐。

| # | tool_key | 模板 | 命中需求（description 要点） | 编排概要 | 卡点 |
|---|---|---|---|---|---|
| P1 | `full_video_creation` | 视频全流程创作 | 复杂视频：多输出单元、需规划 | understand(有素材)→draft_spec→draft_script→draft_storyboard→锚点图(t2i/i2i×N)→单元帧(i2i×N)→单元视频(i2v×N)→tts+music→assemble | 规格、故事板、锚点图 |
| P2 | `quick_video` | 快速出视频 | 一句话/一张图直出短视频 | write_prompt→(t2i→)i2v | 无 |
| P3 | `narrated_video` | 图文解说视频 | 讲解/科普/带货 | draft_script→配图(t2i×N)→tts→assemble | 脚本 |
| P4 | `quick_image` | 快速出图 | 单张/几张图 | write_prompt→t2i/i2i | 无 |
| P5 | `image_series` | 一致性图集 | 同一主体多张图 | 锚点(t2i)→确认→i2i×N | 锚点 |
| P6 | `music_generation` | 配乐生成 | BGM/纯音乐 | write_prompt→generate_music | 无 |
| P7 | `voice_production` | 配音制作 | 文本变语音 | draft_script(可跳)→tts×N→assemble(多段) | 无 |

生长规则：模板跟能力池走——格子工具实现后对应模板入库（generate_song→歌曲创作、audio_driven_video→数字人/MV 等）；动态编排的成功产物按 M6 沉淀持续扩充库。
