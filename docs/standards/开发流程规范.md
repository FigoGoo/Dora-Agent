# 开发流程规范

状态：active  
owner：主控 Codex 汇总维护  
适用范围：产品、智能体微服务、业务微服务、前端、测试协作  
更新时间：2026-06-22  

## 核心原则

- 开发以产品体验设计师产出的《产品系统设计》为起点。
- 产品文档未确认 Done，不允许进入正式开发。
- 产品 Done 后，工程 subagent 必须先完成需求映射矩阵，再进入契约或代码开发。
- 开发按功能点拆分，按功能点提交，避免大而杂的提交。
- 当前阶段只做本地开发：本地环境为 macOS + Docker；线上环境为 CentOS 8 单机，发布与告警/SLO 上线后再完善。

## 流程总览

```text
用户需求
  -> 产品体验设计师输出《产品系统设计》
  -> 产品确认 Done
  -> 产品体验设计师输出项目开发计划
  -> 主控 Codex 拆解功能点和 owner
  -> 工程 subagent 输出需求映射矩阵
  -> 契约先行：Thrift/RPC、API、AG-UI、Agent 数据模型
  -> 本地开发：macOS + Docker
  -> 按功能点提交与 PR
  -> 本地验证与缺陷修复
  -> 测试阶段：QA 执行浏览器/RPC/数据库测试
  -> 每次迭代准备 SQL 脚本
```

## Product Done Gate

《产品系统设计》必须满足以下条件才算 Done：

- 文档状态标记为 `product_status: Done`。
- 产品目标、非目标、用户角色、用户流程、功能清单、Agent 能力清单、页面结构、UI/UE、业务规则、异常场景、验收标准完整。
- 已列出交付给 Go Eino 智能体微服务架构工程师、前端开发工程师、业务微服务后端工程师、测试工程师的说明。
- 验收标准可测试，不是模糊描述。
- 主控 Codex 没有未关闭的产品阻塞问题。
- 已通过 `docs/templates/项目开发计划模板.md` 输出功能点、owner、里程碑和依赖。

未满足 Done Gate 时，工程 subagent 只能做澄清、方案评估、技术预研或文档评审，不能进入正式代码开发。

## Definition of Ready

功能点进入开发前必须满足：

- 产品文档已 Done。
- 功能点 owner 明确。
- 需求映射矩阵已完成。
- 涉及跨服务交互时，RPC / API / AG-UI 契约已确认 owner。
- 涉及数据库时，已确认业务 DB 或 Agent DB 边界。
- 涉及智能体时，已确认 Agent 配置项、Tool、Skill、Workflow、Graph、Memory、Interrupt / Resume 影响。
- 涉及本地运行时，`.env.example` 中的配置项足够支撑本地 macOS Docker 开发。

## Definition of Done

功能点完成必须满足：

- 代码或文档变更只覆盖该功能点范围。
- 相关契约、数据模型、配置、SQL 脚本和文档已同步。
- 本地验证已完成，无法验证时说明原因和风险。
- GitHub 提交按功能点拆分。
- PR 描述包含产品映射、契约影响、SQL 脚本、验证结果和风险。
- 若存在缺陷，已按 owner 回派或记录在缺陷报告。

## 契约优先

- 智能体微服务和业务微服务之间先定义 Thrift / Kitex RPC 契约。
- 智能体微服务和前端之间先定义 HTTP API 与 AG-UI 事件协议。
- Agent Runtime 数据先定义 Agent 领域数据模型。
- 未文档化的字段、事件、错误码、状态机不得直接进入代码。

## 本地开发

- 本地开发环境：macOS + Docker。
- 本地依赖：PostgreSQL、etcd、必要的 mock 服务通过 Docker 启动。
- `.env.example` 是配置项事实源，开发者复制为本地 `.env.local` 或等价文件。
- 本地环境只用于开发和联调，不代表线上部署形态。

## 线上约束

- 线上目标环境：CentOS 8 单机。
- 上线前不在本规范中定义 CI/CD、SLO、告警和自动发布流程。
- 上线后再补充发布、监控、告警、SLO、备份恢复等规范。

## SQL 脚本

- 每个迭代如果涉及数据库结构或初始数据，必须准备 SQL 脚本。
- 当前阶段本地可以使用 golang-migrate 执行；线上由用户手动迁移。
- 每个 SQL 脚本必须包含 up/down 或明确不可回滚说明。
- 业务库 SQL 和 Agent 领域库 SQL 必须分目录保存。
