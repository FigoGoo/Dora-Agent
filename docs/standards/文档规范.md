# 文档规范

状态：active
owner：文档与契约责任域
更新时间：2026-06-28
适用范围：docs 下所有项目文档

## 文档分类

- 当前事实源索引：当前任务默认读取入口和文档状态说明。
- 技术设计文档：架构、后端、前端、主题样式、CI/CD、功能迭代和技术决策。
- PRD：产品目标、用户流程、Agent 能力和验收标准。
- 产品系统设计：产品与需求责任域交付给工程责任域的产品目标、用户流程、Agent 能力、页面结构、业务规则、异常场景、验收标准和交付说明。
- ADR：架构或技术决策。
- RPC 契约：智能体微服务和业务微服务边界。
- API 契约：前端和智能体微服务边界。
- AG-UI 事件协议：智能体微服务和前端实时事件边界。
- Agent 领域数据模型：Agent Runtime 数据库表和状态机。
- 测试报告：测试范围、结果、风险和上线建议。
- 缺陷报告：可复现问题和证据。
- 产品设计交接规范：见 `docs/standards/产品设计交接与开发约束规范.md`。
- 开发流程规范：产品 Done Gate、按功能点开发、本地开发和 SQL 准备规则。
- GitHub 协作规范：分支、提交、PR、评审和合并规则。
- 安全规范：认证、授权、租户、Secret、Agent Tool 和日志脱敏规则。
- 本地开发与配置规范：macOS + Docker 本地环境和 env 模板。
- 迭代 SQL 脚本规范：本地 SQL 和上线手动迁移准备规则。
- TOS 对象存储规范：公共桶、object key、上传签名、前端直传和公开媒体快照规则。
- 阶段交付归档：已完成阶段的交付范围、验收结论、历史设计入口和遗留风险。

## 状态

- draft：草案，未作为实现依据。
- review：评审中，可用于讨论、澄清和补充，不能直接进入正式代码开发。
- active：当前有效事实源。
- archived：历史阶段归档，只能按需追溯，不再承接新迭代。
- deprecated：已废弃，必须说明替代文档或废弃原因。

## 必填元信息

- owner：负责人或维护角色。
- 更新时间：使用 `YYYY-MM-DD`。
- 适用范围：说明服务、模块、页面、接口或数据范围。
- 相关代码路径：列出主要目录或文件。
- 相关契约：列出 RPC、API、AG-UI 或数据模型文档。

## 内容要求

- 背景：为什么需要这份文档。
- 目标：要达成的结果。
- 非目标：明确本次不覆盖的内容。
- 约束：服务边界、技术、权限、事务、事件、数据或测试约束。
- 方案：结构化说明设计、契约、流程或验证。
- 风险：列出影响范围和缓解方式。
- 验收标准：必须可验证。

## 更新规则

- Codex 执行任务时先读 `AGENTS.md` 和 `docs/current/README.md`，再按任务读取对应目录索引。
- 契约变更先更新文档，再改实现。
- 代码行为与文档不一致时，必须同步修正文档或代码。
- 文档更新要说明影响范围。
- 新功能开发前，如果缺产品、技术、契约或测试口径，先补对应文档。
- 开发完成后，必须在对应文档或测试报告中记录实现状态、验证命令、证据路径、遗留风险和后续动作。
- 阶段推进、临时计划和里程碑过程记录不得继续追加到 current 文档；完成后归入 `docs/releases/**` 或 `docs/archive/**`。

## 最新文档读取规则

- `docs/current/README.md` 是最新文档入口。
- 各目录 `README.md` 是目录内文档状态和关注级别的事实源。
- `active` 文档优先于 `archived`、`deprecated` 文档。
- 同一主题存在多份文档时，优先读取更新时间较新且状态为 `active` 的文档；如仍冲突，按 `docs/current/README.md` 和目录 README 的指向为准。
- `code-plan/**` 是第一阶段服务端开发历史归档，默认不作为后续迭代事实源。

## 开发前设计文档规则

- 产品变更：先更新 PRD、产品设计文档或 `docs/product/iterations/**`。
- 后端功能：先更新 `docs/technical/backend/features/**`、`docs/technical/iterations/**`、RPC/API 契约、数据模型、SQL 和测试用例。
- 前端功能：先更新 `docs/technical/frontend/features/**`、页面状态、API/AG-UI 消费规则、主题样式规范和测试用例。
- 跨服务能力：先更新 `docs/contracts/**`，再改 Agent 服务、业务服务或前端实现。
- 规范变更：先更新 `docs/standards/**`，并在相关 Skill 或 README 中同步读取方式。

## 开发完成记录规则

- 实现完成后，在相关技术设计或契约文档中记录当前实现状态。
- 功能迭代完成后，在 `docs/technical/iterations/**` 记录验证证据，并把阶段性结论写入 `docs/releases/**`。
- 测试完成后，在测试报告或测试用例文档中记录命令、结果、证据路径和未执行原因。
- 功能验收完成后，阶段性结论写入 `docs/releases/**`。
- 过期设计必须标记 `archived` 或 `deprecated`，并指向替代文档。

## 废弃规则

- deprecated 文档不得删除，除非文档与契约责任域明确清理。
- 废弃文档必须指向替代文档或说明历史原因。
- archived 文档不再承接新内容；如果需要继续使用，必须迁移成新的 active 文档。
