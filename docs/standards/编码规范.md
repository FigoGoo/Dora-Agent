# 编码规范

状态：active  
owner：主控 Codex 汇总维护  
更新时间：2026-06-25
适用范围：所有服务、前端、测试和工程化代码  

## 通用编码原则

- 修改必须服务于当前任务，不做无关重构。
- 优先遵守现有项目结构、命名、错误处理和测试模式。
- 代码需要表达清晰边界：前端、智能体微服务、业务微服务、测试各司其职。
- 任何跨服务交互先看契约，再写实现。
- 不确定依赖 API 时，查当前依赖、官方文档或已有示例，不臆造。
- 开发必须遵守产品 Done Gate、GitHub 按功能点提交、安全规范、本地配置规范和迭代 SQL 脚本规范。

## 后端通用约束

- 数据库 schema 创建一律不添加数据库级外键约束，也不通过关联键表达表关联；跨表一致性通过业务流程、幂等键、唯一约束、必要索引、application/domain 校验和测试保证。
- 尽量避免在 `for` 循环中逐条查询数据库或 RPC；优先批量查询、`IN` 查询、JOIN、预加载或批量 RPC。
- 列表查询必须分页，默认 10 条每页。
- 入参、出参和展示对象优先使用 DTO 包装；DTO 按领域和业务场景划分，避免复用大对象、ORM 对象或跨场景通用对象。

## 服务边界

- 前端负责页面、组件、状态、HTTP/SSE/WebSocket、AG-UI 消费和用户交互。
- 智能体微服务负责 Eino Agent Runtime、Agent API、AG-UI 事件生产、TurnLoop、RPC client 和 Agent 领域数据。
- 业务系统微服务负责业务规则、业务数据库、事务、权限、领域模型、RPC server 和业务审计。
- 测试工程负责浏览器、AG-UI、Agent API、RPC、Agent DB、业务 DB 验证。

## 错误处理

- 错误必须可分类：用户输入、业务规则、权限、RPC、工具、模型、系统。
- 对外错误码稳定，内部错误保留 trace_id 或 run_id 便于排查。
- 可恢复错误要说明恢复方式；不可恢复错误要中止并输出失败状态。

## 日志规范

- 服务端日志应包含 trace_id、tenant_id、user_id、session_id、run_id、task_id、tool_call_id、rpc_method、latency、error_code 中的相关字段。
- 日志不得泄露密钥、敏感个人信息或完整业务机密数据。
- Agent Runtime 日志要能还原 run 生命周期和工具调用链路。

## 测试要求

- Go 代码运行 gofmt 和 go test。
- 前端代码运行项目约定的 lint、typecheck、unit test 或 browser test。
- RPC 变更补充 contract test。
- AG-UI 变更验证事件类型、payload、顺序、幂等和重连。
- 数据库变更分别验证 Agent 领域数据库和业务数据库。

## 文档同步要求

- RPC、API、AG-UI、Agent 数据模型、状态机和错误码变更必须同步文档。
- 架构性决策使用 ADR。
- 测试结论使用测试报告模板，缺陷使用缺陷报告模板。

## 禁止事项

- 禁止大规模无关重构。
- 禁止跨服务直接访问数据库。
- 禁止智能体微服务直接写业务数据库。
- 禁止业务微服务承担 Eino Agent 编排。
- 禁止前端发明未文档化 API 或 AG-UI 字段。
- 禁止把契约写死在代码里而不文档化。
- 禁止产品文档未 Done 时进入正式代码开发。
- 禁止提交真实 env、密钥、Token、AK/SK。
