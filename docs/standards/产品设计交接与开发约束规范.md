# 产品设计交接与开发约束规范

状态：active  
owner：主控 Codex 汇总维护  
适用范围：产品体验设计师、Go Eino 智能体微服务架构工程师、前端开发工程师、业务微服务后端工程师  
更新时间：2026-06-22  

## 目标

让产品体验设计师产出的《产品系统设计》成为工程开发的上游事实源，确保智能体微服务、前端和业务微服务按同一份产品目标、流程、规则和验收标准开发。

## 产品设计文档最低要求

产品体验设计师输出的《产品系统设计》必须至少包含：

- 产品目标。
- 用户角色。
- 使用场景。
- 用户流程。
- 功能清单。
- Agent 能力清单。
- 页面结构。
- UI/UE 规范。
- 交互状态。
- 业务规则。
- 异常场景。
- 非目标。
- 验收标准。
- 对 Go Eino 智能体微服务架构工程师、前端开发工程师、业务微服务后端工程师的交付说明。

## 工程读取顺序

三个工程角色 owner 在开发前按以下顺序读取产品设计文档：

1. 元信息：确认文档状态、owner、更新时间、适用范围和关联文档。
2. 产品目标：确认本次开发要达成的用户结果。
3. product_status：确认产品文档是否 Done；未 Done 时不能进入正式开发。
4. 非目标：确认本次不做什么，避免擅自扩展。
5. 用户角色：确认权限、租户、使用入口和差异化体验。
6. 用户流程：确认主路径、分支路径、人工确认、失败重试和完成条件。
7. 功能清单：确认功能拆分和 owner。
8. Agent 能力清单：确认 Agent 输入、输出、工具、人工确认、产物和限制。
9. 页面结构与 UI/UE 规范：确认前端页面、组件、状态和事件展示。
10. 业务规则与异常场景：确认业务事实、权限、事务、错误码、审计和边界。
11. 验收标准：将每条验收标准映射为测试或可验证检查。

## 需求映射矩阵

每个工程角色 owner 开发前必须产出自己的需求映射矩阵：

| 产品设计条目 | 工程解释 | 产出物 | owner | 契约/测试 |
| --- | --- | --- | --- | --- |
| 产品目标 / 用户流程 / Agent 能力 / 业务规则 / 验收标准 | 对本角色的具体含义 | API、RPC、AG-UI、组件、领域模型、测试等 | 对应 owner | 关联文档或测试 |

没有映射到工程产物的产品条目必须说明原因：不在当前范围、需要澄清、由其他工程领域 负责，或被非目标排除。

## Go Eino 智能体微服务架构工程师读取规则

重点读取：

- Agent 能力清单。
- 用户流程中的 Agent 交互、人工确认、长任务和失败恢复。
- 页面状态对 AG-UI 事件的要求。
- 业务规则中需要 RPC 调用的业务能力。
- 验收标准中涉及 Agent 行为、事件流、Tool 调用、产物、任务状态和可观测性的条目。

必须产出：

- Eino 能力选型：Agent / Graph / Workflow / Tool / Skill / Retriever / Memory / Callback / Interrupt / TurnLoop。
- Agent API 设计。
- AG-UI 事件生产方案。
- RPC client 调用需求。
- Agent 领域数据模型影响。
- Agent 行为测试和事件流测试映射。

禁止：

- 直接把业务规则写进 Agent 代替业务服务。
- 直接访问业务数据库。
- 在产品未定义人工确认时自行新增确认流程；如有安全必要，必须回报主控 Codex。

## 前端开发工程师读取规则

重点读取：

- 用户角色和用户流程。
- 页面结构、导航结构、UI/UE 规范。
- 交互状态：loading、empty、error、success、streaming、interrupt、resume。
- Agent 流式输出、Tool 调用、人工确认和任务状态展示。
- 验收标准中涉及用户可见行为的条目。

必须产出：

- 页面/路由/组件映射。
- 状态管理设计。
- HTTP API 依赖。
- AG-UI 事件消费矩阵。
- 用户交互测试和浏览器验证点。

禁止：

- 发明后端字段或 AG-UI 事件字段。
- 用前端逻辑替代业务规则。
- 在契约未定义时自行猜测事件 payload。

## 业务微服务后端工程师读取规则

重点读取：

- 用户角色、权限和租户规则。
- 业务规则、异常场景和审计要求。
- 用户流程中涉及业务事实变化的步骤。
- Agent 能力清单中需要业务服务支持的读写能力。
- 验收标准中涉及业务数据、权限、事务、错误码和审计的条目。

必须产出：

- 业务领域模型影响。
- Kitex RPC server 能力清单。
- Thrift DTO、错误码、权限上下文、幂等键和审计字段建议。
- PostgreSQL / GORM / migration 影响。
- 业务服务测试和 contract test 映射。

禁止：

- 承担 Eino Agent 编排。
- 暴露业务数据库表操作作为 RPC 方法。
- 为了 Agent 调用便利破坏领域模型。

## 冲突与澄清规则

遇到以下情况必须停止开发并回报主控 Codex：

- 产品目标、用户流程、业务规则或验收标准相互矛盾。
- 产品设计缺少关键字段、权限、状态、异常场景或验收标准。
- Agent 能力需要业务事实写入，但没有对应 RPC 契约。
- 前端需要展示的 AG-UI 事件未定义。
- 业务服务认为产品流程违反业务规则、权限或事务一致性。
- 任一工程角色 owner 需要跨越自己的服务边界才能完成需求。

## 开发约束

- 产品设计文档是需求事实源，但不能替代 RPC、API、AG-UI、数据模型和测试文档。
- 工程实现必须保持从产品条目到代码、契约、测试的可追溯关系。
- 产品设计未达到 `product_status: Done` 时，只能做澄清、方案设计、原型或技术预研，不能进入正式代码开发。
- 产品设计变更后，工程角色 owner 必须重新检查需求映射矩阵和受影响契约。

## 输出格式

三个工程角色 owner 在开始实现前输出：

```text
产品设计读取结果
需求映射矩阵
本角色负责范围
依赖其他工程领域的内容
需要新增或变更的契约
测试与验收映射
冲突、缺口或待确认事项
```
