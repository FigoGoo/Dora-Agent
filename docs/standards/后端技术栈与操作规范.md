# 后端技术栈与操作规范

状态：active  
owner：主控 Codex 汇总维护  
适用范围：业务微服务后端工程师、Go Eino 智能体微服务架构工程师  
更新时间：2026-06-25

## 目标

统一 Go 后端技术栈和操作入口，避免业务微服务、智能体微服务在 RPC、IDL、数据库、迁移、配置发现、日志和测试上各自选型。

## 技术栈事实源

| 领域 | 技术框架 | 使用边界 | 官方/操作文档 |
| --- | --- | --- | --- |
| RPC | Kitex | 智能体微服务与业务微服务之间的 RPC；业务服务作为 Kitex server，智能体服务作为 Kitex client | https://www.cloudwego.io/docs/kitex/getting-started/tutorial/ |
| IDL | Thrift | RPC 契约唯一 IDL；`.thrift` 文件是跨服务契约事实源 | https://thrift.apache.org/docs/idl |
| HTTP | Gin | 智能体服务承载 Agent API、SSE、WebSocket、AG-UI HTTP 入口时使用；业务服务仅在确需直接暴露 HTTP 时使用 | https://gin-gonic.com/en/docs/ |
| DB | PostgreSQL | 业务数据库和 Agent 领域数据库均使用 PostgreSQL，但必须物理或逻辑隔离 | https://www.postgresql.org/docs/current/ |
| DAO | GORM | Go 服务访问 PostgreSQL 的 DAO/Repository 实现基础；业务规则不得写进 DAO | https://gorm.io/docs/ |
| Migration | golang-migrate | 数据库 schema 变更工具；禁止用运行时 AutoMigrate 替代正式 migration | https://github.com/golang-migrate/migrate |
| 配置/服务发现 | etcd | 服务发现、Kitex registry/resolver、非敏感运行配置 | https://etcd.io/docs/ 与 https://www.cloudwego.io/docs/kitex/tutorials/third-party/service_discovery/etcd/ |
| 日志 | 火山引擎日志服务 | 统一采集结构化业务日志、Agent Runtime 日志和必要 Trace；敏感字段脱敏 | https://www.volcengine.com/docs/6470/812322 |
| 测试 | Go testing + testify | 单元、集成、契约、RPC client/server、DAO、Agent 行为测试 | https://github.com/stretchr/testify |

## CloudWeGo 官方文档优先

- Kitex、Thrift、etcd registry/resolver、代码生成和 client/server option 必须优先遵循 CloudWeGo 官方文档。
- 具体操作要求见 `docs/standards/CloudWeGo开发操作规范.md`。
- 如本项目规范与 CloudWeGo 官方文档冲突，以官方文档为准，并回到主控 Codex 更新规范。

## RPC：Kitex

- 业务微服务默认暴露 Kitex server。
- 智能体微服务通过 Kitex client 调用业务服务。
- 服务名、方法名、请求 DTO、响应 DTO、错误码、超时、重试、幂等、权限上下文和审计字段必须先进入 RPC 契约文档。
- Kitex client 必须设置 context timeout；写操作必须携带幂等键。
- 服务注册与发现使用 etcd，优先使用 Kitex etcd registry/resolver 扩展。
- 不允许绕过 Kitex RPC 直接访问对方数据库。

## IDL：Thrift

- RPC 契约使用 `.thrift` 文件维护。
- Thrift 文件建议放在 `api/thrift/**`，由 Go Eino 智能体微服务架构工程师提出调用需求，业务微服务后端工程师确认业务语义。
- Thrift service 表达业务能力，不表达数据库表 CRUD。
- Thrift struct 字段必须稳定；新增字段保持向后兼容；删除或改语义必须走版本化。
- IDL 变更后必须重新生成 Kitex 代码，并补充 contract test。

## HTTP：Gin

- 智能体微服务需要暴露 Agent API、SSE、WebSocket 或 AG-UI HTTP 入口时，使用 Gin。
- 业务微服务默认不直接暴露 HTTP；只有明确产品或系统集成需要时才使用 Gin。
- Gin handler 只做 transport 层：参数绑定、鉴权上下文提取、错误映射和响应包装。
- 业务逻辑放 application/domain，数据库访问放 repository/DAO。

## DB：PostgreSQL

- 业务服务使用业务数据库；智能体服务使用 Agent 领域数据库。
- 两类数据库必须隔离，至少使用独立 schema；生产环境优先独立 database 或实例策略。
- 数据库 schema 创建一律不添加数据库级外键约束，也不通过关联键表达表关联；跨表一致性通过业务流程、幂等键、唯一约束、必要索引、application/domain 校验和测试保证。
- 事务边界必须在 application 层明确。
- 写操作必须考虑唯一约束、幂等键、锁或事务隔离级别。
- 索引设计以查询路径为依据，避免无依据索引。
- 列表查询必须分页，默认 10 条每页；分页参数、排序字段和 page_size 上限必须进入 RPC、API 或数据模型文档。

## DAO：GORM

- GORM 只作为 DAO/Repository 实现基础，不承载业务规则。
- 查询必须传递 context。
- 事务使用 `db.WithContext(ctx).Transaction(...)` 或项目既有等价模式。
- 禁止在服务启动或请求路径中使用 GORM AutoMigrate 改生产 schema。
- 复杂查询优先保持可读；必要时使用 SQL，但仍由 repository 封装。
- 尽量避免在 `for` 循环中逐条查询数据库或 RPC；优先批量查询、`IN` 查询、JOIN、预加载或批量 RPC。

## Migration：golang-migrate

- 每次 schema 变更创建成对的 `up.sql` / `down.sql`。
- migration 文件必须进入代码评审，说明影响表、索引、锁风险、回滚方案。
- 业务库 migration 由业务微服务后端工程师负责。
- Agent 领域库 migration 由 Go Eino 智能体微服务架构工程师负责。
- 本项目工程化阶段不创建真实业务 migration；只有实现阶段在明确需求后创建。

建议命令形态：

```bash
migrate create -ext sql -dir migrations -seq <change_name>
migrate -path migrations -database "$DATABASE_URL" up
migrate -path migrations -database "$DATABASE_URL" down 1
```

## 配置/服务发现：etcd

- Kitex 服务注册、服务发现使用 etcd。
- 非敏感运行配置可以放 etcd；密钥、AK/SK、数据库密码不得放入普通 etcd 配置路径。
- etcd key 必须有命名空间，例如 `/dora/<env>/<service>/<config_key>`。
- 客户端必须设置连接超时，复用 client，退出时关闭。
- 配置变更需要审计，关键配置变更需要回滚方案。

## 日志：火山引擎日志服务

- 服务日志统一输出结构化字段，至少包含 `trace_id`、`service`、`env`、`tenant_id`、`user_id`、`session_id`、`run_id`、`rpc_method`、`error_code`、`latency_ms` 中适用字段。
- 智能体服务必须补充 `agent_session_id`、`agent_run_id`、`tool_call_id`、`graph_node`、`event_id` 等 Agent Runtime 字段。
- 业务服务必须补充 `business_action`、`resource_id`、`idempotency_key`、`audit_result` 等业务审计字段。
- 接入火山引擎日志服务时，AK/SK、Endpoint、Region、Token 等凭证从环境变量或安全配置读取，不写入代码。
- Trace 接入参考火山引擎“通过 OpenTelemetry SDK 写入数据”文档；日志写入和消费参考火山引擎日志服务 Go SDK 文档。
- 日志不得输出密钥、完整 Token、原始密码、支付凭证、未脱敏个人敏感信息。

## 测试：Go testing + testify

- 所有 Go 测试以标准 `testing` 为入口。
- 使用 `testify/assert` 做非阻断断言，使用 `testify/require` 做前置条件或失败后无法继续的断言。
- RPC 契约测试覆盖 Kitex client/server 的请求、响应、错误码、超时、权限、幂等。
- DAO 测试覆盖 PostgreSQL 约束、事务和关键查询。
- Agent 服务测试覆盖 Tool、Graph/Workflow、TurnLoop、AG-UI 事件和 RPC mock。

## 两类后端 工程角色分工

### 业务微服务后端工程师

- 负责 Kitex server、Thrift 业务能力定义确认、业务 PostgreSQL、GORM repository、业务 migration、业务日志、业务测试。
- 负责按业务领域和场景拆分请求 DTO、响应 DTO 和展示 DTO，避免复用业务大对象。
- 不负责 Eino 编排、AG-UI 事件协议和 Agent 领域数据库。

### Go Eino 智能体微服务架构工程师

- 负责 Kitex client、Thrift 调用需求提出、Agent PostgreSQL、GORM repository、Agent 领域 migration、Agent Runtime 日志、Agent 测试。
- 负责为 Tool、RPC client 和 Agent 领域数据使用场景化 DTO，并提出批量查询或分页 RPC 需求。
- 不负责业务数据库写入、业务规则最终解释和业务 Kitex server 实现。

## 检查表

- [ ] 是否使用 Kitex + Thrift 作为 RPC/IDL。
- [ ] 是否只在需要 HTTP 入口时使用 Gin。
- [ ] 是否使用 PostgreSQL + GORM，并保持业务 DB 与 Agent DB 边界。
- [ ] 是否没有创建数据库级外键约束。
- [ ] 列表查询是否分页且默认 10 条每页。
- [ ] 是否避免 `for` 循环逐条查询数据库或 RPC。
- [ ] DTO 是否按领域和业务场景拆分。
- [ ] 是否使用 golang-migrate 管理 schema 变更。
- [ ] 是否使用 etcd 做服务发现和非敏感配置。
- [ ] 是否按火山引擎日志服务要求输出结构化日志并保护敏感信息。
- [ ] 是否使用 Go testing + testify 完成匹配范围测试。
