# 编码规范执行

## 目标

确保每次代码修改都符合服务边界、契约、错误处理、日志、测试和文档同步要求。

## 使用场景

- 任意代码修改任务。
- 代码评审或修复缺陷。
- 新增 API、RPC、AG-UI、Agent DB 或前端事件消费。

## 输入

- 变更需求和相关文件。
- docs/standards 下的通用规范和技术域规范。
- RPC、API、AG-UI、Agent 数据模型文档。

## 文档使用规则

- 开始任务先读 `AGENTS.md` 和 `docs/current/README.md`，再按任务读取 `docs/technical/README.md`、`docs/product/README.md`、`docs/contracts/README.md`、`docs/standards/README.md` 或 `docs/test/README.md`。
- 开发前若缺产品目标、技术设计、契约、数据模型、SQL 或测试口径，先补对应 `draft` 或 `active` 文档，不直接从历史归档派生实现。
- 开发前需要设计时，按 `docs/templates/` 下对应模板新建或更新 `docs/technical/**`、`docs/product/**`、`docs/contracts/**` 或 `docs/test/**`，状态先用 `draft` 或 `review`；达到当前事实源条件后再改为 `active`。
- 旧文档不再承接当前迭代、内容被字段级事实源替代、包含已完成阶段计划或与当前代码冲突时，标记为 `archived` 或 `deprecated`，移动到 `docs/archive/**` 或 `docs/releases/**`，并在当前入口写明替代文档。
- 开发中新增或改变 RPC、API、AG-UI、数据模型、SQL、配置、权限、错误码或测试夹具时，同步更新对应文档和目录 README。
- 开发完成后，在相关设计、契约或测试报告中记录实现状态、验证命令、证据路径、未执行原因、遗留风险和后续动作。
- `code-plan/**`、`docs/releases/**`、`docs/archive/**` 默认只用于历史追溯；需要复用其中结论时，先迁移到当前 active 文档。

## 执行流程

1. 读取规范：先读 `docs/standards/编码规范.md` 和对应领域规范。
2. 判断修改范围：列出必须修改的文件，不做无关重构。
3. 检查服务边界：智能体服务不写业务库，业务服务不做 Eino 编排，前端不发明后端字段。
4. 检查 RPC 契约：跨服务调用必须有服务名、方法、DTO、错误码、超时、重试、幂等、权限、审计和 contract test。
5. 检查 AG-UI 契约：事件类型、payload、顺序、幂等、断线重连和未知事件兼容必须明确。
6. 检查错误处理：错误可分类、可观察、可恢复性清楚。
7. 检查日志：包含 trace_id、session_id、run_id、tool_call_id、rpc_method 等排障字段。
8. 检查测试：根据变更运行单元、集成、契约、浏览器或数据库验证。
9. 检查文档同步：契约、数据模型和状态变更同步 docs。
10. 输出规范符合性说明。

## 输出

- 修改范围。
- 服务边界检查。
- RPC 契约检查。
- AG-UI 契约检查。
- 错误处理和日志检查。
- 测试结果。
- 文档同步说明。
- 规范符合性结论。

## 检查表

- [ ] 是否读取规范。
- [ ] 是否只修改必要文件。
- [ ] 是否没有跨服务直接访问数据库。
- [ ] 是否没有伪造契约字段。
- [ ] 是否有错误处理和日志。
- [ ] 是否有测试或替代验证说明。
- [ ] 是否同步文档。

## 注意事项

- 规范执行不是格式化清单，必须结合当前变更判断风险。
- 不能用“暂时方便”绕过 RPC 或 AG-UI 契约。
- 无法验证时要说明阻塞原因和风险。
