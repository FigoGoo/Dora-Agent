# 编码规范执行

## 目标

确保每次代码修改都符合服务边界、契约、错误处理、日志、测试和文档同步要求。

## 使用场景

- 任意 subagent 修改代码。
- 代码评审或修复缺陷。
- 新增 API、RPC、AG-UI、Agent DB 或前端事件消费。

## 输入

- 变更需求和相关文件。
- docs/standards 下的通用规范和技术域规范。
- RPC、API、AG-UI、Agent 数据模型文档。

## 执行流程

1. 读取规范：先读 `docs/standards/编码规范.md` 和对应领域规范。
2. 判断修改范围：列出必须修改的文件，不做无关重构。
3. 检查服务边界：智能体服务不写业务库，业务服务不做 Eino 编排，前端不发明后端字段。
4. 检查 RPC 契约：跨服务调用必须有服务名、方法、DTO、错误码、超时、重试、幂等、权限、审计和 contract test。
5. 检查 AG-UI 契约：事件类型、payload、顺序、幂等、断线重连和未知事件兼容必须明确。
6. 检查错误处理：错误可分类、可观察、可恢复性清楚。
7. 检查日志：包含 trace_id、session_id、run_id、tool_call_id、rpc_method 等排障字段。
8. 检查测试：根据变更运行单元、集成、契约、浏览器或数据库验证。
9. 检查文档同步：契约、数据模型和状态变更同步 docs。
10. 输出规范符合性说明。

## 输出

- 修改范围。
- 服务边界检查。
- RPC 契约检查。
- AG-UI 契约检查。
- 错误处理和日志检查。
- 测试结果。
- 文档同步说明。
- 规范符合性结论。

## 检查表

- [ ] 是否读取规范。
- [ ] 是否只修改必要文件。
- [ ] 是否没有跨服务直接访问数据库。
- [ ] 是否没有伪造契约字段。
- [ ] 是否有错误处理和日志。
- [ ] 是否有测试或替代验证说明。
- [ ] 是否同步文档。

## 注意事项

- 规范执行不是格式化清单，必须结合当前变更判断风险。
- 不能用“暂时方便”绕过 RPC 或 AG-UI 契约。
- 无法验证时要说明阻塞原因和风险。
