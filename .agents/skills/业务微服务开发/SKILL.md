# 业务微服务开发

## 目标

在业务系统微服务内实现业务规则和数据一致性，同时为智能体服务提供稳定 RPC 能力。

## 使用场景

- 新增或修改业务 RPC server。
- 实现领域模型、业务规则、数据库写入或权限校验。
- 为 Agent 调用设计业务能力。

## 输入

- PRD、RPC 契约和业务规则。
- 数据库模型、权限模型和错误码规范。
- 服务端测试要求。

## 执行流程

1. RPC server：按契约实现服务名、方法、请求、响应、错误码和超时。
2. 领域模型：让业务概念、状态和约束在业务服务中表达。
3. 业务规则：业务服务拥有最终解释权，不把规则转移到 Agent。
4. 数据库读写：通过 repository 或项目既有模式访问业务数据库。
5. 事务：写操作必须明确事务边界和回滚条件。
6. 权限：在业务服务内校验用户、租户、角色和资源权限。
7. 错误码：稳定、可文档化、可测试，不泄露内部实现。
8. 审计：记录操作者、动作、资源、结果、trace_id 和时间。
9. 测试：覆盖单元、集成、事务、权限、幂等、错误码和 contract test。

## 输出

- RPC server 设计或实现说明。
- 领域模型和业务规则。
- 数据库读写与事务边界。
- 权限、错误码、审计。
- 测试结果。

## 检查表

- [ ] 是否遵守 RPC 契约。
- [ ] 是否保持业务服务拥有业务规则。
- [ ] 写操作是否有事务和幂等设计。
- [ ] 权限是否在业务服务校验。
- [ ] 是否没有暴露数据库内部结构给智能体服务。
- [ ] 是否覆盖服务端测试。

## 注意事项

- 业务微服务不负责 Eino Agent 编排。
- 不为方便 Agent 调用破坏领域模型。
- 不把 AG-UI 事件协议放入业务服务职责。
