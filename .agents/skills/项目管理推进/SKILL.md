# 项目管理推进

## 目标

让产品体验设计师不仅产出产品设计，还能把产品文档转化为可推进的开发计划和交付节奏。

## 使用场景

- 《产品系统设计》进入 Review 或 Done 阶段。
- 需要安排功能点开发顺序、owner 和里程碑。
- 需要确认产品 Done Gate。
- 需要协调 Eino、前端、业务后端和测试的交付顺序。
- 需要跟踪阻塞项、风险和验收状态。

## 输入

- 用户需求。
- 《产品系统设计》。
- `docs/standards/开发流程规范.md`。
- `docs/standards/产品设计交接与开发约束规范.md`。
- `docs/templates/项目开发计划模板.md`。
- 各 subagent 的需求映射矩阵、契约文档和测试报告。

## 执行流程

1. 确认产品状态：Draft、Review、Done；未 Done 时列出缺口。
2. 拆分功能点：按用户流程、Agent 能力、页面、业务规则和验收标准拆分。
3. 排列优先级：优先主路径、基础契约、核心 Agent 能力、本地可验证路径。
4. 指定 owner：每个功能点绑定产品、Eino、前端、业务后端或测试 owner。
5. 定义里程碑：说明每个里程碑的目标、交付物、Done 条件和风险。
6. 建立依赖：标出 RPC、API、AG-UI、Agent 数据模型、SQL、前端页面之间的依赖。
7. 推进交付：要求工程 subagent 输出需求映射矩阵后再开发。
8. 管理阻塞：产品缺口、契约冲突、跨边界问题必须回到主控 Codex。
9. 验收闭环：按产品验收标准检查工程产出和测试报告。

## 输出

- 产品状态和 Done Gate 结论。
- 项目开发计划。
- 功能点拆分。
- owner 和优先级。
- 里程碑。
- 依赖关系。
- 阻塞项和风险。
- 验收状态。

## 检查表

- [ ] 产品文档是否达到 Done。
- [ ] 功能点是否可独立开发和提交。
- [ ] owner 是否清晰。
- [ ] 依赖是否明确。
- [ ] 是否覆盖主路径。
- [ ] 是否要求工程 subagent 输出需求映射矩阵。
- [ ] 是否有验收标准和测试入口。

## 注意事项

- 项目管理推进不替代工程设计。
- 产品未 Done 时不能推动正式开发。
- 功能点拆分要服务于 GitHub 按功能点提交和 PR。
- 计划变更必须同步产品文档和开发任务。
