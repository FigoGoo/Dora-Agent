# 项目管理推进

## 目标

让产品与需求责任域不仅产出产品设计，还能把产品文档转化为可推进的开发计划和交付节奏。

## 使用场景

- 《产品系统设计》进入 Review 或 Done 阶段。
- 需要安排功能点开发顺序、责任域和里程碑。
- 需要确认产品 Done Gate。
- 需要协调 Eino、前端、业务服务和测试的交付顺序。
- 需要跟踪阻塞项、风险和验收状态。

## 输入

- 用户需求。
- 《产品系统设计》。
- `docs/standards/开发流程规范.md`。
- `docs/standards/产品设计交接与开发约束规范.md`。
- `docs/templates/项目开发计划模板.md`。
- 各责任域的需求映射矩阵、契约文档和测试报告。

## 文档使用规则

- 开始任务先读 `AGENTS.md` 和 `docs/current/README.md`，再按任务读取 `docs/technical/README.md`、`docs/product/README.md`、`docs/contracts/README.md`、`docs/standards/README.md` 或 `docs/test/README.md`。
- 开发前若缺产品目标、技术设计、契约、数据模型、SQL 或测试口径，先补对应 `draft` 或 `active` 文档，不直接从历史归档派生实现。
- 开发前需要设计时，按 `docs/templates/` 下对应模板新建或更新 `docs/technical/**`、`docs/product/**`、`docs/contracts/**` 或 `docs/test/**`，状态先用 `draft` 或 `review`；达到当前事实源条件后再改为 `active`。
- 旧文档不再承接当前迭代、内容被字段级事实源替代、包含已完成阶段计划或与当前代码冲突时，标记为 `archived` 或 `deprecated`，移动到 `docs/archive/**` 或 `docs/releases/**`，并在当前入口写明替代文档。
- 开发中新增或改变 RPC、API、AG-UI、数据模型、SQL、配置、权限、错误码或测试夹具时，同步更新对应文档和目录 README。
- 开发完成后，在相关设计、契约或测试报告中记录实现状态、验证命令、证据路径、未执行原因、遗留风险和后续动作。
- `code-plan/**`、`docs/releases/**`、`docs/archive/**` 默认只用于历史追溯；需要复用其中结论时，先迁移到当前 active 文档。

## 执行流程

1. 确认产品状态：Draft、Review、Done；未 Done 时列出缺口。
2. 拆分功能点：按用户流程、Agent 能力、页面、业务规则和验收标准拆分。
3. 排列优先级：优先主路径、基础契约、核心 Agent 能力、本地可验证路径。
4. 指定责任域：每个功能点绑定产品与需求、Agent 服务、前端、业务服务或测试与验收责任域。
5. 定义里程碑：说明每个里程碑的目标、交付物、Done 条件和风险。
6. 建立依赖：标出 RPC、API、AG-UI、Agent 数据模型、SQL、前端页面之间的依赖。
7. 推进交付：要求工程责任域输出需求映射矩阵后再开发。
8. 管理阻塞：产品缺口、契约冲突、跨边界问题必须回到文档与契约责任域。
9. 验收闭环：按产品验收标准检查工程产出和测试报告。

## 输出

- 产品状态和 Done Gate 结论。
- 项目开发计划。
- 功能点拆分。
- 责任域和优先级。
- 里程碑。
- 依赖关系。
- 阻塞项和风险。
- 验收状态。

## 检查表

- [ ] 产品文档是否达到 Done。
- [ ] 功能点是否可独立开发和提交。
- [ ] 责任域是否清晰。
- [ ] 依赖是否明确。
- [ ] 是否覆盖主路径。
- [ ] 是否要求工程责任域输出需求映射矩阵。
- [ ] 是否有验收标准和测试入口。

## 注意事项

- 项目管理推进不替代工程设计。
- 产品未 Done 时不能推动正式开发。
- 功能点拆分要服务于 GitHub 按功能点提交和 PR。
- 计划变更必须同步产品文档和开发任务。
