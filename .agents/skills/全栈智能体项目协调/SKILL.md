# 全栈项目协调

## 目标

让跨责任域任务在多微服务协作中保持边界清晰、契约优先、结果可合并。

## 使用场景

- 任务涉及产品、Agent Runtime、业务服务、前端或测试中的两个以上领域。
- 需要按责任域拆解任务、识别依赖或安排交付顺序。
- RPC 契约、AG-UI 事件协议或文件责任域发生冲突。
- 需要汇总多个责任域的结论并决定下一步。

## 输入

- 用户需求或缺陷描述。
- AGENTS.md 中的架构边界和责任域。
- 相关 docs/standards 规范。
- 已有 PRD、ADR、RPC、API、AG-UI、数据模型或测试文档。

## 文档使用规则

- 开始任务先读 `AGENTS.md` 和 `docs/current/README.md`，再按任务读取 `docs/technical/README.md`、`docs/product/README.md`、`docs/contracts/README.md`、`docs/standards/README.md` 或 `docs/test/README.md`。
- 开发前若缺产品目标、技术设计、契约、数据模型、SQL 或测试口径，先补对应 `draft` 或 `active` 文档，不直接从历史归档派生实现。
- 开发前需要设计时，按 `docs/templates/` 下对应模板新建或更新 `docs/technical/**`、`docs/product/**`、`docs/contracts/**` 或 `docs/test/**`，状态先用 `draft` 或 `review`；达到当前事实源条件后再改为 `active`。
- 旧文档不再承接当前迭代、内容被字段级事实源替代、包含已完成阶段计划或与当前代码冲突时，标记为 `archived` 或 `deprecated`，移动到 `docs/archive/**` 或 `docs/releases/**`，并在当前入口写明替代文档。
- 开发中新增或改变 RPC、API、AG-UI、数据模型、SQL、配置、权限、错误码或测试夹具时，同步更新对应文档和目录 README。
- 开发完成后，在相关设计、契约或测试报告中记录实现状态、验证命令、证据路径、未执行原因、遗留风险和后续动作。
- `code-plan/**`、`docs/releases/**`、`docs/archive/**` 默认只用于历史追溯；需要复用其中结论时，先迁移到当前 active 文档。

## 执行流程

1. 判断任务规模：单文件小改直接处理；跨责任域任务先列清责任域、依赖和验收口径；存在契约冲突时先做契约任务。
2. 按责任域拆解：产品与需求、Agent 服务、业务服务、前端、测试与验收、文档与契约分别明确输入、输出和边界。
3. 产品设计交接：产品文档进入 Done 后，受影响的工程责任域先输出需求映射矩阵，再进入契约或代码开发。
4. 依赖控制：只并行推进互不依赖的任务；共享契约、共享数据库或共享测试夹具必须先对齐事实源。
5. RPC 契约冲突处理：Agent 服务提出业务能力需求，业务服务确认业务语义、权限、错误码和事务边界，文档与契约责任域维护 `docs/contracts/**`。
6. AG-UI 事件冲突处理：Agent 服务确认事件来源和顺序，前端确认渲染状态和容错，文档与契约责任域维护协议文档。
7. 合并结果：按产品设计映射、边界、契约、测试、风险五类归档，冲突项标明责任域和后续处理方式。
8. 缺陷回派：测试与验收责任域提供证据，按责任域定位并修复。

## 输出

- 任务拆解和责任域分工说明。
- 产品设计交接和需求映射矩阵。
- RPC / AG-UI / 文件责任域冲突结论。
- 合并后的修改文件清单。
- 测试与验证结果。
- 风险、待确认事项和下一步。

## 注意事项

- 文档与契约责任域负责维护跨服务契约，不替业务服务定义业务事实。
- RPC 契约先于跨服务实现。
- AG-UI 事件协议先于前端消费和智能体事件生产。
- Agent 领域数据库与业务数据库必须分开验证。
