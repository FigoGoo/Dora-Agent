# 全栈智能体项目协调

## 目标

让主控 Codex 在多微服务、多 subagent 协作中保持边界清晰、契约优先、结果可合并。

## 使用场景

- 任务涉及产品、Agent Runtime、业务服务、前端或测试中的两个以上领域。
- 需要判断是否调度 subagent。
- RPC 契约、AG-UI 事件协议或文件 ownership 发生冲突。
- 需要汇总多个 subagent 的输出并决定下一步。

## 输入

- 用户需求或缺陷描述。
- AGENTS.md 中的架构边界和 ownership。
- 相关 docs/standards 规范。
- 已有 PRD、ADR、RPC、API、AG-UI、数据模型或测试文档。

## 执行流程

1. 判断任务规模：单文件小改由主控直接处理；跨领域任务拆给对应 subagent；存在契约冲突时先做契约任务。
2. 按 ownership 调度：产品找产品体验设计师，Eino/Agent 找 Go Eino 智能体微服务架构工程师，业务规则找业务微服务后端工程师，前端找前端开发工程师，验证找浏览器、RPC 与数据库测试工程师。
3. 产品设计交接：产品体验设计师产出《产品系统设计》后，要求 Eino、前端、业务后端三个 subagent 先输出需求映射矩阵，再进入契约或代码开发。
4. 控制并发：只并发互不依赖的任务，最多 5 个 subagent，subagent 不递归调度。
5. RPC 契约冲突处理：智能体侧提出业务能力需求，业务服务侧确认业务语义、权限、错误码和事务边界，主控维护 docs/contracts。
6. AG-UI 事件冲突处理：智能体侧确认事件来源和顺序，前端侧确认渲染状态和容错，主控维护协议文档。
7. 合并结果：按产品设计映射、边界、契约、测试、风险五类归档，冲突项回派给 owner。
8. 缺陷回派：浏览器、RPC 与数据库测试工程师提供证据，主控按 ownership 指派修复。

## 输出

- 任务拆解和 subagent 调度说明。
- 产品设计交接和需求映射矩阵。
- RPC / AG-UI / 文件 ownership 冲突结论。
- 合并后的修改文件清单。
- 测试与验证结果。
- 风险、待确认事项和下一步。

- [ ] 是否避免 subagent 职责重叠。
- [ ] 是否完成产品设计交接和需求映射。
- [ ] 是否明确 RPC 契约 owner。
- [ ] 是否明确 AG-UI 事件协议 owner。
- [ ] 是否遵守文件 ownership。
- [ ] 是否汇总了每个 subagent 的结论和证据。
- [ ] 是否将缺陷回派给正确 owner。

## 注意事项

- 主控 Codex 负责协调，不替业务服务定义业务事实。
- RPC 契约先于跨服务实现。
- AG-UI 事件协议先于前端消费和智能体事件生产。
- Agent 领域数据库与业务数据库必须分开验证。
