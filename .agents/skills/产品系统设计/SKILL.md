# 产品系统设计

## 目标

把模糊业务想法整理为可交付给后端、智能体、前端和测试的产品设计。

## 使用场景

- 新增或调整 AIGC Agent SaaS 功能。
- 需要定义用户流程、页面范围或验收标准。
- 需要把音乐、图片、视频等生成能力拆成 Agent 能力清单。

## 输入

- 用户需求、业务背景、目标用户。
- 现有 PRD、产品文档和页面设计。
- RPC、API、AG-UI、Agent 能力边界。

## 文档使用规则

- 开始任务先读 `AGENTS.md` 和 `docs/current/README.md`，再按任务读取 `docs/technical/README.md`、`docs/product/README.md`、`docs/contracts/README.md`、`docs/standards/README.md` 或 `docs/test/README.md`。
- 开发前若缺产品目标、技术设计、契约、数据模型、SQL 或测试口径，先补对应 `draft` 或 `active` 文档，不直接从历史归档派生实现。
- 开发前需要设计时，按 `docs/templates/` 下对应模板新建或更新 `docs/technical/**`、`docs/product/**`、`docs/contracts/**` 或 `docs/test/**`，状态先用 `draft` 或 `review`；达到当前事实源条件后再改为 `active`。
- 旧文档不再承接当前迭代、内容被字段级事实源替代、包含已完成阶段计划或与当前代码冲突时，标记为 `archived` 或 `deprecated`，移动到 `docs/archive/**` 或 `docs/releases/**`，并在当前入口写明替代文档。
- 开发中新增或改变 RPC、API、AG-UI、数据模型、SQL、配置、权限、错误码或测试夹具时，同步更新对应文档和目录 README。
- 开发完成后，在相关设计、契约或测试报告中记录实现状态、验证命令、证据路径、未执行原因、遗留风险和后续动作。
- `code-plan/**`、`docs/releases/**`、`docs/archive/**` 默认只用于历史追溯；需要复用其中结论时，先迁移到当前 active 文档。

## 执行流程

1. 明确产品目标：说明用户要完成什么结果。
2. 定义用户角色：区分普通用户、运营、管理员、内容审核等角色。
3. 梳理使用场景：覆盖创建、编辑、生成、确认、失败重试、历史查看。
4. 设计用户流程：用步骤描述入口、关键决策、Agent 交互和完成条件。
5. 建立功能清单：拆分为页面功能、Agent 能力、业务服务能力和测试点。
6. 定义 Agent 能力清单：说明输入、输出、工具使用、人工确认和边界。
7. 写清非目标：排除本期不做的能力。
8. 标出业务规则和异常场景：包括权限、额度、失败、取消、超时和重复提交。
9. 定义验收标准：每项标准必须可测试。
10. 标记产品状态：Draft、Review、Done；只有 Done 才允许工程责任域进入正式开发。

## 输出

- 产品目标。
- 用户角色。
- 使用场景。
- 用户流程。
- 功能清单。
- Agent 能力清单。
- 非目标。
- 业务规则。
- 异常场景。
- 验收标准。
- product_status。
- 对工程责任域的交付说明。

## 检查表

- [ ] 用户目标是否清晰。
- [ ] 用户角色是否覆盖主要权限差异。
- [ ] Agent 能力是否可映射到智能体服务。
- [ ] 业务事实是否交给业务服务处理。
- [ ] 异常场景是否包含失败、取消、超时和重试。
- [ ] 验收标准是否可由测试与验收责任域验证。
- [ ] product_status 是否明确。
- [ ] Done 状态是否满足开发流程规范。

## 注意事项

- 不在产品设计中发明未确认的后端字段。
- 产品设计只定义业务期望，不替 Eino 架构做技术选型。
- 对跨服务能力，用 RPC 契约或 API 契约承接，不写成口头约定。
- 未达到 Done 的产品文档不能作为正式开发依据。
