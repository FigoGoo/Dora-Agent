# EinoGraph工作流设计

## 目标

把确定性、多节点、可分支的 Agent 任务设计为可测试、可观测、可恢复的 Graph 或 Workflow。

## 使用场景

- 生成音乐、图片、视频等任务存在固定阶段。
- 需要节点级状态、事件、错误路径和重试策略。
- 需要判断流程是否封装为 Agent Tool。

## 输入

- 流程目标和产品验收标准。
- 输入结构、输出结构、业务 RPC 能力。
- AG-UI 事件协议和 Agent 状态模型。

## 文档使用规则

- 开始任务先读 `AGENTS.md` 和 `docs/current/README.md`，再按任务读取 `docs/technical/README.md`、`docs/product/README.md`、`docs/contracts/README.md`、`docs/standards/README.md` 或 `docs/test/README.md`。
- 开发前若缺产品目标、技术设计、契约、数据模型、SQL 或测试口径，先补对应 `draft` 或 `active` 文档，不直接从历史归档派生实现。
- 开发前需要设计时，按 `docs/templates/` 下对应模板新建或更新 `docs/technical/**`、`docs/product/**`、`docs/contracts/**` 或 `docs/test/**`，状态先用 `draft` 或 `review`；达到当前事实源条件后再改为 `active`。
- 旧文档不再承接当前迭代、内容被字段级事实源替代、包含已完成阶段计划或与当前代码冲突时，标记为 `archived` 或 `deprecated`，移动到 `docs/archive/**` 或 `docs/releases/**`，并在当前入口写明替代文档。
- 开发中新增或改变 RPC、API、AG-UI、数据模型、SQL、配置、权限、错误码或测试夹具时，同步更新对应文档和目录 README。
- 开发完成后，在相关设计、契约或测试报告中记录实现状态、验证命令、证据路径、未执行原因、遗留风险和后续动作。
- `code-plan/**`、`docs/releases/**`、`docs/archive/**` 默认只用于历史追溯；需要复用其中结论时，先迁移到当前 active 文档。

## 执行流程

1. 定义流程目标：说明 Graph / Workflow 完成什么结果。
2. 定义输入结构：包含用户输入、上下文、权限、配置和幂等键。
3. 定义输出结构：包含最终结果、产物、状态、错误和事件摘要。
4. 列出节点列表：每个节点只做一个清晰步骤。
5. 定义节点输入输出：字段、来源、去向和校验规则。
6. 设计边和分支：说明顺序、条件、并行、回退和终止条件。
7. 设计错误路径：可重试、不可重试、需要人工确认、需要回滚的路径分开。
8. 设计状态读写：只写 Agent 领域状态；业务事实通过 RPC。
9. 设计事件输出：节点开始、完成、失败映射为 AG-UI 事件。
10. 判断是否封装为 Agent Tool：当流程可被上层 Agent 调用且输入输出稳定时封装。
11. 编写测试用例：覆盖正常路径、分支、错误、重试、中断和恢复。

## 输出

- 流程目标。
- 输入结构和输出结构。
- 节点列表。
- 节点输入输出。
- 边和分支。
- 错误路径。
- 状态读写。
- 事件输出。
- 是否封装为 Agent Tool。
- 测试用例。

## 检查表

- [ ] 节点职责是否单一。
- [ ] 输入输出是否明确。
- [ ] 分支和错误路径是否可测试。
- [ ] 状态是否只写 Agent 领域表。
- [ ] 业务事实是否通过 RPC。
- [ ] 节点事件是否能映射到 AG-UI。

## 注意事项

- 不要把开放式推理硬塞进确定性 Graph。
- 不要让 Graph 节点直接操作业务数据库。
- 长任务必须考虑状态持久化和恢复。
