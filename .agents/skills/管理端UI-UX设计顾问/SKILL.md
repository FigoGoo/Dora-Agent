# 管理端UI-UX设计顾问

## 目标

为 Dora-Agent 平台后台建立清晰、克制、高密度、可扫描、可审计、易实现的管理端体验。管理端样式主题明确采用简约风格：白色主题、黑色元素点缀、中性灰层级。管理端是配置、审核、排障和审计工作台，不是用户端创作空间；不得复用 C 端暗色电影级背景、玻璃拟态、作品氛围图、霓虹光效、营销式 Hero 或大面积渐变。

## 使用场景

- 设计或审查 `admin_frontend/**` 的后台页面、App Shell、导航、列表页、详情页、创建/编辑页、审核页、权限页和设置页。
- 设计或优化管理员账号、用户管理、系统 Skill、Skill 审核、模型供应商、模型、Tool、积分、兑换码、精选作品、审计日志等后台模块。
- 设计表格、筛选、高级筛选、分页、排序、列配置、固定列、批量操作、导出和行内操作。
- 设计配置表单、权限矩阵、敏感字段、API Key / Secret 输入、危险操作确认、审计提示和变更记录。
- 输出页面结构、组件规范、交互说明、UX 文案、状态说明、响应式规则、可访问性建议和前端验收标准。

## 不使用场景

- 用户端首页、Agent 创作工作台、作品中心、Prompt 输入器和 C 端创作者工具：改用 `用户端UI-UX设计顾问`。
- 官网、品牌落地页、活动页、宣传页、海报、Banner、Logo、插画或品牌 KV；但管理端页面落地缺少必要 logo、登录背景、空状态图、引导图或动效关键图时，应使用 Image 2 生成克制的首版原创图片资产，并标记为可替换资产。
- 纯 React、Vue、CSS、API 报错修复，且不涉及体验、状态或交互设计。
- 权限、安全、隐私、财务和合规最终判断；只能提出 UI/UX 层面的风险提示、确认和审计展示建议。
- 复制第三方后台一模一样的界面、专有图形、完整页面布局或可识别商业外观。

## 输入

至少需要以下任意一种：

- 页面截图、页面描述、后台菜单结构、当前问题或需要设计的页面名称。
- 业务对象、用户角色、核心任务、表格字段、表单字段、审核流程或权限层级。
- 使用频率、数据量级、操作风险、组件库约束、设备端或需要交付给谁。

信息不足时，先列出关键假设并给出初版方案；只在无法判断核心业务对象或高风险边界时停止追问。追问最多 3 个。

## 文档入口

- 项目文档、契约、规范和测试口径统一以 `docs/` 下内容为准。

## 默认设计方向

- 视觉气质：简约白色主题、黑色元素点缀、中性灰层级、低装饰、高密度、表格优先、边界清晰、状态语义明确。
- 主题主色：以黑色或近黑作为主操作、当前导航、关键强调和可交互焦点；彩色只用于语义状态，不把蓝色作为管理端默认主色。
- 页面骨架：`AdminSideNav 232px/64px + AdminTopBar 56px + AdminPage + PageHeader + FilterBar/ActionBar + DataTable/Form/DetailPanel + Drawer/Modal/Toast`。
- 密度尺寸：页面水平内边距 `24px`，窄屏 `16px`；表格行高默认 `44px`，高信息行 `52px`；Drawer `480px` 到 `720px`；表单主体最大宽度 `960px`。
- 组件气质：卡片圆角不超过 `8px`；表格、筛选栏、表单控件不使用大圆角胶囊；普通卡片少用阴影，浮层才用轻阴影。
- 状态表达：成功绿、警告琥珀、危险红、信息青；状态不得只靠颜色表达，必须有文字或图标辅助。
- 响应式：桌面是主场景；平板折叠导航、减少低优先级列；移动端只保证基础查看、筛选和关键操作，不强行完整搬运复杂配置和批量操作。

## 图片资产缺口处理

- 管理端默认不依赖大面积装饰图；但登录页、品牌标识、空状态、引导说明、导出成功、异步处理中或错误修复引导等场景确实需要图片时，不能用破图、空白、纯渐变或临时色块替代。
- 先检查仓库、设计文档和已有品牌资产；确认缺失后，使用 Image 2 生成原创 bitmap 图片资产，再接入页面实现。
- 生成图片必须保持后台气质：白底或浅灰环境、低饱和、中性、清晰、低装饰，不使用 C 端沉浸式作品氛围、霓虹、玻璃拟态、营销 Hero 或强视觉噪音。
- Logo 或品牌符号只能作为首版可替换资产，不作为最终品牌定稿；文件命名和代码注释应体现 `placeholder`、`generated` 或 `provisional` 语义。
- 动效图需求优先生成关键帧、序列帧、纹理或封面图，再用 CSS、canvas、视频或 Lottie 管线实现动效；后台动效必须短、轻、可跳过，不影响表格和表单效率。
- 生成后必须在前端验收中检查清晰度、裁切、加载体积、浅色背景对比度、文本遮挡和是否破坏高密度后台布局。

## 已知反例沉淀

管理端 UI 审查时必须主动排查以下反复出现的问题。发现任一问题时，先按问题整理输出，再给出修复建议或直接修复。

### 原生控件穿帮

- 只给原生 `select` 加边框、圆角和图标是不够的；展开后仍会出现系统原生下拉菜单，和管理端主题不一致。
- 筛选栏、分页、表单中的状态、类型、页大小等选项集，优先使用自绘 `button + listbox` / Dropdown / AdminSelect，而不是裸 `select`。
- 如果保留原生日期、时间或文件选择能力，必须用管理端外壳包住默认控件，并在浏览器里检查普通态、focus 态和展开/选择态；视觉不一致时改用自绘组件。
- 验收时不能只看静态截图，必须点击展开下拉菜单，确认菜单背景、选中态、hover、阴影、边框和字体都符合管理端 token。

### 筛选栏操作层级错误

- `FilterBar` 里的“清除”“重置”“更多筛选”等是低强调辅助操作，应使用小尺寸 ghost/text 样式，不得做成像主操作一样的大块按钮。
- 筛选栏内搜索、状态、时间、操作者等控件高度必须一致；按钮不得撑高筛选栏或造成文本垂直不齐。
- 已选筛选条件可以显式展示，但不得让清除按钮比筛选条件更抢眼。

### Focus 和输入框反馈过重

- 输入框、搜索框、Select、DatePicker 的 focus 使用 `border-strong + 低透明 focus ring`；不得使用厚重黑色阴影、默认蓝色轮廓或浏览器原生高亮。
- 搜索框必须用 `:focus-within` 或等效机制让外层控件显示 focus，而不是只让内部无边框 input 获得焦点。
- 错误态优先用红色边框、字段下错误文案和必要图标；不要把 focus、error、hover 三种状态混成同一种强阴影。

### Modal / Drawer 浮层瑕疵

- Modal、Drawer 的关闭按钮不得因为 Tooltip、overflow、z-index 或 transform 出现黑条、裁切、遮挡或位置漂移。
- 浮层 header 内的 icon button 若使用 Tooltip，必须验证 Tooltip 不被 `overflow:hidden` 或浮层边界裁切；不确定时关闭按钮不使用 Tooltip，只保留 `aria-label`。
- Modal footer 保持右对齐：取消/次要操作在左，保存/确认在右；危险确认必须在确认弹层中保留对象、影响、原因和审计提示。
- Drawer 详情优先用于列表上下文内只读查看，不应像表格一样横向溢出。

### 详情信息原始输出

- Drawer、详情页、审计摘要不得直接 `JSON.stringify` 后端对象给管理员看。
- 对象和数组必须结构化成 key-value、分组、列表或摘要；空数组和空对象显示 `-` 或“暂无记录”。
- 长字段名、ID、trace_id 和嵌套字段必须允许换行或复制；不得撑破 Drawer 宽度。
- 继续遵守敏感信息边界：Secret、完整兑换码、用户私有资产、会话、黑板、生成记录、提示词、供应商原始响应不得明文展示。

### 导航 active 误判

- 左侧导航必须保证同一时刻只有语义上正确的菜单 active；父路径和子路径相似时必须检查是否需要精确匹配。
- 典型反例：`/admin/models/providers` 不得同时高亮“模型供应商”和“模型管理”；`/admin` 首页必须用精确匹配。
- 审查时至少点击一个父子路径相邻的模块，确认 active 背景没有重复。

### 危险操作视觉层级

- 表格行内危险操作使用低强调 danger ghost/text 按钮，避免每行都出现高饱和红色块。
- 真正执行危险操作的确认弹窗按钮才使用强 danger 样式。
- 危险操作不能只靠 Toast；必须有 preview/影响说明、原因输入、确认、结果反馈和审计提示。

## 分析流程

1. 业务对象：识别页面管理的是管理员、用户、Skill、模型、Tool、积分、兑换码、作品、审计日志或配置。
2. 用户角色：识别超级管理员、管理员、运营、审核、客服、财务、企业管理员等角色的可见信息和操作权限。
3. 核心任务：排序查询、筛选、创建、编辑、审核、导出、配置、排障、审计和高风险处理。
4. 信息架构：组织导航、页面分区、关键字段、主操作、次操作、行内操作、批量操作和危险操作。
5. 表格体验：检查字段顺序、关键列、状态列、操作列、列宽、固定列、列配置、分页、排序、加载、空、错和批量操作。
6. 筛选体验：检查搜索、常用筛选、高级筛选、时间范围、状态快捷筛选、已选条件、一键清除和保存视图。
7. 表单体验：检查字段分组、必填、默认值、校验、错误、只读、禁用、提交中、保存失败保留输入、取消确认和变更记录。
8. 权限与风险：检查无权限隐藏/禁用策略、敏感字段遮罩、危险操作二次确认、原因输入、影响说明、可恢复性和审计提示。
9. 状态与反馈：覆盖 loading、empty、filtered_empty、error、success、confirming、permission_denied、partial_success 和 async_processing。
10. 图片资产：检查必要 logo、登录背景、空状态图、引导图和动效关键图是否存在；缺失时触发 Image 2 生成并列入资产清单。
11. 动态交互：必须检查下拉展开态、Modal/Drawer 打开态、Tooltip、focus、hover、禁用态、错误态和确认态，不只看默认静态页面。
12. 响应式：至少检查桌面 `1440px` 和平板 `768px`；确认侧栏折叠、筛选栏换行、表格横滚、浮层宽度和文本不重叠。
13. 交付验收：把重要建议转为设计师可画、前端可实现、测试可验证的描述。

## 严重程度

- Critical：阻断核心后台任务，或导致严重权限、财务、数据、审计、不可逆误操作风险。
- High：显著降低运营效率，增加误操作概率，或让审核、配置、导出、状态判断难以完成。
- Medium：造成中等效率问题、字段理解问题、状态不一致、表单分组不清或视觉层级混乱。
- Low：圆角、边框、hover、空状态文案、图标一致性等细节优化。

## 浏览器验收要求

当任务允许使用浏览器或用户提供截图时，管理端 UI/UX 审查必须至少做以下验收：

1. 打开目标页面默认态，检查布局、密度、颜色、表格和筛选栏。
2. 点击所有可疑下拉、日期、分页和更多菜单，检查展开态是否仍符合后台主题。
3. Tab 或点击进入输入框、搜索框、下拉、按钮，检查 focus 态是否轻量且清晰。
4. 打开新增/编辑 Modal、详情 Drawer、危险确认弹窗，检查关闭按钮、Tooltip、遮罩、阴影、footer、滚动和文本换行。
5. 打开至少一个详情 Drawer，确认没有 raw JSON、隐私字段明文、横向撑破和不可复制长 ID。
6. 点击父子路径相邻的导航项，确认 active 状态唯一且准确。
7. 在 `1440px` 和 `768px` 视口下复查筛选栏换行、表格横滚、侧栏折叠和浮层宽度。
8. 若已实现代码，补充或建议回归测试：自绘 Select 打开 listbox 且不渲染原生 `select`；Drawer 不展示 raw JSON；导航 active 不重复；高风险操作确认链路完整。

## 注意事项

- 管理端的“简约”不是空白变多，而是减少装饰、保留必要信息、稳定层级、提升扫描效率。
- 没有数据时，不声称“一定提升效率”“一定降低误操作”或“一定降低运营成本”；改为“预计降低操作成本，建议通过任务完成时间和运营人员可用性测试验证”。
- 危险操作不能只靠 Toast；必须在确认、结果反馈和审计提示中形成闭环。
- 后台字段、权限和审计结构以后端契约为准；UI 方案只定义展示与交互，不替代契约。
