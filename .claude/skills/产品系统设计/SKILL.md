# 产品系统设计

## 目标

把模糊业务想法整理为可交付给后端、智能体、前端和测试的产品设计。

## 使用场景

- 新增或调整 AIGC Agent SaaS 功能。
- 需要定义用户流程、页面范围或验收标准。
- 需要把音乐、图片、视频等生成能力拆成 Agent 能力清单。

## 输入

- 用户需求、业务背景、目标用户。
- 现有 PRD、产品文档和页面设计。
- RPC、API、AG-UI、Agent 能力边界。

## 执行流程

1. 明确产品目标：说明用户要完成什么结果。
2. 定义用户角色：区分普通用户、运营、管理员、内容审核等角色。
3. 梳理使用场景：覆盖创建、编辑、生成、确认、失败重试、历史查看。
4. 设计用户流程：用步骤描述入口、关键决策、Agent 交互和完成条件。
5. 建立功能清单：拆分为页面功能、Agent 能力、业务服务能力和测试点。
6. 定义 Agent 能力清单：说明输入、输出、工具使用、人工确认和边界。
7. 写清非目标：排除本期不做的能力。
8. 标出业务规则和异常场景：包括权限、额度、失败、取消、超时和重复提交。
9. 定义验收标准：每项标准必须可测试。
10. 标记产品状态：Draft、Review、Done；只有 Done 才允许工程 subagent 进入正式开发。

## 输出

- 产品目标。
- 用户角色。
- 使用场景。
- 用户流程。
- 功能清单。
- Agent 能力清单。
- 非目标。
- 业务规则。
- 异常场景。
- 验收标准。
- product_status。
- 对工程 subagent 的交付说明。

## 检查表

- [ ] 用户目标是否清晰。
- [ ] 用户角色是否覆盖主要权限差异。
- [ ] Agent 能力是否可映射到智能体服务。
- [ ] 业务事实是否交给业务服务处理。
- [ ] 异常场景是否包含失败、取消、超时和重试。
- [ ] 验收标准是否可由测试工程师验证。
- [ ] product_status 是否明确。
- [ ] Done 状态是否满足开发流程规范。

## 注意事项

- 不在产品设计中发明未确认的后端字段。
- 产品设计只定义业务期望，不替 Eino 架构做技术选型。
- 对跨服务能力，用 RPC 契约或 API 契约承接，不写成口头约定。
- 未达到 Done 的产品文档不能作为正式开发依据。
